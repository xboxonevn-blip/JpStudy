## Parity Gaps

| Area | Evidence | Risk |
|---|---|---|
| ThemeData completeness | `AppTheme.light` defines navigation bar, filled button, outlined button, input decoration, and icon theme; `AppTheme.dark` does not mirror most of these | Material default drift |
| Default mode | provider starts at `ThemeMode.light` and does not support `ThemeMode.system` | no OS dark preference |
| Design Lab | screen body uses fixed light gradient `F9FAFF/F3F6FF/EFFBFF` | dark mode bypass |
| Grammar repair prompts | `multiple_choice_widget.dart` hardcodes light prompt surfaces like `FFFFFCF6`, `FFFFFBFA`, `FFFFFAF3` | bright islands in dark mode |
| Hardcoded color volume | broad grep found `413` hardcoded color/theme hits across `43` files | audit debt |
| Test coverage | only one route-level dark visual probe was found | unknown route parity |

## Caveats

- Hardcoded `Colors.white` is often intentional on dark/gradient hero surfaces, so grep volume is not a defect count.
- Static audit cannot prove final screenshots. Primary route screenshots in dark mode are still needed before claiming visual parity.
- Design Lab may be internal, but it still shows that feature screens can bypass palette tokens entirely.
