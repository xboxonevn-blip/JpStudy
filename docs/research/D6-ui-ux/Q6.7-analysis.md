# D6.Q6.7 Analysis - Dark Mode Parity

## E6.7 Result

Dark mode is functional at the shell/theme-token layer, and the existing focused tests pass. It is not parity-ready for beta claims because dark `ThemeData` is less complete than light `ThemeData`, route-level test coverage is narrow, and several feature screens still hardcode light surfaces.

## Findings

1. `H6.7.1` supported. `MaterialApp.router` wires `darkTheme` and `themeMode`; Me settings can persist dark/light mode.
2. `H6.7.2` supported for sampled tokens. Existing tests passed, and Q6.5 already found dark core tokens mostly safer than light muted tokens.
3. `H6.7.3` supported. Light theme configures navigation bar, button, input, and icon themes more completely than dark theme.
4. `H6.7.4` supported. Grammar repair prompts and Design Lab contain fixed light surfaces; broad color grep found `413` matches in `43` files.
5. `H6.7.5` supported. Tests validate theme wiring and one visual probe, not the actual primary route family in dark mode.

## Production Impact

Dark mode can be offered as an experimental/user setting, but should not be described as polished parity. A learner using dark mode may see the shell, many shared cards, and palette-backed surfaces correctly, then hit a bright prompt/card island or default Material component that does not match the rest of the UI.

The highest-risk parity areas are:

1. Grammar repair prompts with fixed light surfaces.
2. Design Lab fixed light background, if kept user-visible.
3. Default Material inputs/buttons/chips under dark theme because dark `ThemeData` lacks light-theme component parity.
4. Untested primary routes: Home, Practice, Kanji Hub, Grammar, Vocab, JLPT, Learn/Test.

## Red Team

Steelman opposite: hardcoded whites and light colors are not automatically bugs. Several are intentional white text on dark gradients, brand logos, shadows, or celebratory dialogs. The passing dark probe also shows core palette contrast is sound. That lowers severity, but does not remove parity risk because fixed pale card surfaces and incomplete dark component themes can still produce jarring route-level differences.

## Decision

Do not block all beta work on dark mode. Treat it as a hardening track:

1. Mirror the missing light-theme component families into `AppTheme.dark`.
2. Convert fixed light prompt/card surfaces to palette-aware tokens, starting with Grammar repair prompts.
3. Keep intentional white-on-gradient cases, but document them as exceptions.
4. Add dark-mode widget/screenshot tests for the primary route family before claiming parity.
5. Consider `ThemeMode.system` after parity tests exist; otherwise defaulting to light is safer.

## Action

Next implementation should be `Q6.7b`:

1. Add dark equivalents for `navigationBarTheme`, `filledButtonTheme`, `outlinedButtonTheme`, `inputDecorationTheme`, and `iconTheme`.
2. Update Grammar repair prompt colors to derive from `AppThemePalette` or brightness-aware schemes.
3. Add focused dark widget tests for at least Home, Practice, Grammar repair prompt, Kanji Hub, and Vocab review.
4. Only after route tests pass, expose system-theme mode or consider defaulting to system.
