# D6.Q6.3 Hypotheses - Loading State Audit

## H6.3.1 - Spinners are the dominant loading pattern

Prediction: most async loading branches use `CircularProgressIndicator`, with little or no skeleton/shimmer usage. Effect size: spinner matches outnumber skeleton/shimmer matches by at least 20:1.

Falsifier: skeleton or structured placeholder components are used broadly across primary modules.

## H6.3.2 - Optional dashboard loading often collapses

Prediction: secondary home/profile cards frequently use `SizedBox.shrink()` for loading to avoid layout noise.

Falsifier: optional dashboard cards usually render compact loading placeholders.

## H6.3.3 - Progress indicators mix async loading and learner progress

Prediction: `LinearProgressIndicator` is used for study progress/challenges as well as occasional loading, so the component alone does not communicate "loading".

Falsifier: linear indicators are reserved for one meaning.

## H6.3.4 - Loading copy and semantics are sparse

Prediction: most loading states show bare spinners without localized labels or semantic live-region copy.

Falsifier: sampled loaders consistently include descriptive copy such as "Loading lessons..." in the active language.

## H6.3.5 - Practice/session loading should be stricter than dashboard loading

Prediction: primary practice flows need visible blocking loaders, while optional dashboard panels can stay quiet if the page still has useful content.

Falsifier: all modules can safely use the same loading treatment.
