# D6.Q6.2 Analysis - Empty State Audit

## E6.2 Result

Empty states exist and many are tested, but the system is inconsistent. The strongest empty states use `AppFeatureCard` with a concrete recovery action, while many secondary/home widgets silently disappear with `SizedBox.shrink()`. A reusable `EmptyStateWidget` exists in `lib/core/widgets`, but no feature uses it.

## Findings

1. `H6.2.1` supported. `EmptyStateWidget` has zero feature usages; sampled empty states are mostly local classes or inline branches.
2. `H6.2.2` supported. At least `11 / 20` sampled states lack a visible CTA. This is acceptable for positive "all clear" states, but weak for missing content or filtered results.
3. `H6.2.3` supported. Copy is mixed: some states use `AppLanguage`, some use local `_tr` helpers, and some use local switch statements. Vietnamese is present in most sampled visible states, but not centrally governed.
4. `H6.2.4` supported. Visual styles include `AppFeatureCard`, `HomeSurface` panel, bare centered icon/text, bare text, local compact state, and hidden widget.
5. `H6.2.5` supported. Daily plan, weakness radar, weekly challenge, personal bests, and challenge history use `SizedBox.shrink()` for some empty/loading/error cases, hiding potentially useful state.

## Production Impact

For beta, this is not a launch blocker by itself, because primary study flows usually avoid blank crashes and several empty states are covered by tests. It is a trust problem when users hit content gaps: some modules explain the absence and offer a route forward, while others simply vanish or say "No lessons" without a next step.

The biggest product risk is silent disappearance on the home/dashboard surfaces. If daily plan or weakness radar is empty because upstream state failed, the learner sees a less capable home screen instead of a recoverable state.

## Red Team

Steelman opposite: hiding optional dashboard cards reduces clutter and may be correct when there is genuinely no due work. That is valid for positive all-clear states. It is not valid for load/error states or primary content absence, where users need to know whether there is nothing to do, nothing loaded, or something broken.

## Decision

Do not force every empty state into one widget. Define two categories:

1. Primary empty states must be visible, explanatory, localized, and include a CTA when recovery exists.
2. Optional dashboard absence may collapse only when data loaded successfully and the absence is a positive/non-actionable state.

## Action

Next implementation should be `Q6.2b`:

1. Move `EmptyStateWidget` or a replacement into the active shared UI layer and use it in new primary flows.
2. Add a small `OptionalPanelState` convention for home widgets: loaded-empty can shrink, loading/error should show compact status in debug/beta or at least log telemetry.
3. Add reset/filter CTAs to filtered empty states, especially Library and Grammar search.
4. Convert bare `Center(Text(loadErrorLabel))` paths to `ErrorStateWidget` with retry where a retry action exists.
5. Keep positive empty states like Mistake Bank all-clear, but add a small next-practice CTA only if analytics shows users stall there.
