# D6.Q6.7 Hypotheses - Dark Mode Parity

## H6.7.1 - Dark mode is wired through the app shell

Prediction: `MaterialApp.router` uses `darkTheme` and `themeMode`, and a user setting can persist dark mode.

Falsifier: dark theme exists only as a theme object and is not connected to app state.

## H6.7.2 - Core dark palette contrast is acceptable

Prediction: `AppThemePalette.dark` has strong enough `ink`, `primary`, and semantic colors on dark surfaces for core text.

Falsifier: the existing dark-mode probe or token tests fail normal text contrast.

## H6.7.3 - Dark ThemeData is thinner than light ThemeData

Prediction: `AppTheme.dark` omits several component themes configured in `AppTheme.light`, so default Material components may not match visual language.

Falsifier: dark and light themes configure the same component families.

## H6.7.4 - Hardcoded light surfaces create screen-level parity gaps

Prediction: hardcoded `Colors.white` and light `Color(0xFFF...)` surfaces remain in feature screens and bypass `AppThemePalette`.

Falsifier: hardcoded colors are limited to icons/logos/overlays where light-on-dark is intentional.

## H6.7.5 - Existing dark tests are useful but too narrow

Prediction: tests prove theme wiring and one probe, not dark-mode parity across primary learning routes.

Falsifier: there are dark-mode widget/screenshot tests for the major route families.
