# D6.Q6.3 Analysis - Loading State Audit

## E6.3 Result

Loading states are functional but inconsistent. The app mostly uses bare centered `CircularProgressIndicator` branches, while a few newer surfaces use clear localized loading cards. Optional dashboard widgets sometimes collapse during loading, which keeps the home/profile screens quiet but can hide upstream slowness or failure.

## Findings

1. `H6.3.1` supported. `CircularProgressIndicator` appears `69` times across `53` files. Skeleton/shimmer search found only one structured skeleton class, the progress coach board.
2. `H6.3.2` supported. Six home/me loading branches return `SizedBox.shrink()`, including daily plan, weekly challenge, weakness radar, personal bests, and challenge history.
3. `H6.3.3` supported. `LinearProgressIndicator` appears `32` times across `26` files and is mainly used for learner progress, quiz/practice progress, streak/progress stats, or coach skeleton status. It cannot be read as "loading" without label context.
4. `H6.3.4` supported. Loading copy is sparse in older blocking routes, but strong localized examples exist in Kanji Hub, JLPT Mock Pro, JLPT prep widgets, and Progress Coach. Search has useful loading copy but it is hardcoded English.
5. `H6.3.5` supported. Practice/session routes need visible blocking loaders; optional panels can stay quieter only when surrounding content still gives the learner something useful to do.

## Production Impact

This is not a crash-level blocker, but it affects trust during first-run and low-data states. A learner who sees a bare spinner cannot tell whether app data, local content, network state, or their selected JLPT lane is loading. A learner whose dashboard cards disappear cannot distinguish "nothing due" from "feature unavailable".

The highest-risk cases are primary practice/review routes and top-level hubs because they block the next action. Optional dashboard cards are lower risk, but their hidden loading pattern should not be reused for primary content.

## Red Team

Steelman opposite: adding copy to every spinner can make short loads feel heavier and clutter screens that usually resolve quickly. This is true for sub-second async branches and optional panels below useful content. It does not justify unlabeled blocking routes that cover the whole screen or hidden loading on panels that represent core daily guidance.

## Decision

Do not build a generic shimmer system now. Define a small loading-state policy:

1. Primary blocking routes: visible centered/card loader with localized label when the route can exceed a short delay or loads learner-specific content.
2. Practice/session startup: visible loader is required; use copy that names the session bank, review queue, or JLPT lane.
3. Optional dashboard panels: may collapse only for loaded-empty positive states; loading/error should use a compact placeholder, beta debug hint, or telemetry.
4. `LinearProgressIndicator`: reserve for learner progress unless it is paired with explicit loading copy.
5. New async cards should follow the Kanji Hub/JLPT/Progress Coach examples rather than adding another bare spinner.

## Action

Next implementation should be `Q6.3b`:

1. Add a small shared `LoadingStateWidget` or document a `LoadingState` pattern beside `ErrorStateWidget`.
2. Convert one primary route family first: vocab catalog/search or grammar practice, because these affect first-session trust.
3. Localize Search loading copy through `AppLanguage` or a feature copy helper.
4. Add compact loading/error treatment to daily plan, weekly challenge, and weakness radar before beta telemetry.
5. Add widget tests only for converted surfaces; do not snapshot every spinner.
