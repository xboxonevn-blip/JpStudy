# D6.Q6.1 Analysis - Card Pattern Audit

## E6.1 Result

Card design is directionally coherent but not systematized. Newer hub surfaces share `AppSectionCard`, `AppFeatureCard`, and `AppCompactRow`; home and practice flows also have legitimate specialized surfaces. The risk is not total visual chaos; it is undocumented parallel conventions.

## Findings

1. `H6.1.1` supported. At least `10 / 20` sampled surfaces are custom or use a non-`AppSectionCard` surface (`HomeSurface`, `GrammarPracticePanel`, Material `Card`, task-state `Container`/`Ink`).
2. `H6.1.2` supported. Sampled radius values include `10`, `12`, `16`, `18`, `20`, `22`, `24`, and `28`. `AppSpacing` labels `16` as cards/dialogs and `20` as panels/modals, while `AppSectionCard` uses `28`.
3. `H6.1.3` supported. Home has a parallel `HomeSurface.softPanel()` system with radius `24`, one shadow, and caller-defined padding. It is used by dense home widgets like Weakness Radar and Weekly Challenge.
4. `H6.1.4` supported. Practice modules diverge for good reasons: learn/grammar/test surfaces encode prompt, choice, correctness, and state feedback. They should not blindly become `AppFeatureCard`.
5. `H6.1.5` supported. CTA placement has at least five patterns: bottom button wrap, section-header action, trailing arrow row, full-card tap, and no-CTA progress card.

## Production Impact

The current system is acceptable for beta if future edits follow a taxonomy instead of inventing more one-off containers. Learners will likely notice inconsistency most when moving from polished hub cards to task/practice screens or older Material `Card` surfaces.

The highest-leverage cleanup is documentation plus opportunistic convergence, not a broad refactor. Changing all radii/shadows now would risk layout regressions without evidence that visual inconsistency is a top beta blocker.

## Red Team

Steelman opposite: different modules may benefit from different surface languages. A daily dashboard, a quiz prompt, and a flashcard should not be identical. That is valid; the defect is that the taxonomy is implicit, so future contributors cannot tell intentional specialization from drift.

## Decision

Do not do a mass card refactor. Adopt a three-family surface taxonomy:

1. Hub/marketing/info surfaces: `AppSectionCard` / `AppFeatureCard`.
2. Dense home dashboard surfaces: `HomeSurface.softPanel`.
3. Active practice/task surfaces: `QuestionPromptCard`, `QuestionChoiceTile`, `GrammarPracticePanel`, and state-specific custom panels.

## Action

Next implementation should be `Q6.1b`:

1. Add a short `docs/design/ui-surface-taxonomy.md` with allowed card families, radius/shadow defaults, and CTA placement rules.
2. Align `AppSpacing` radius comments with the actual taxonomy, or rename comments so `AppSectionCard` radius `28` is not contradictory.
3. Convert stray Material `Card` usage in `PracticeTestDashboard` only when touching that screen for another reason.
4. Add a lightweight lint/review checklist: new feature surfaces must use one of the three families or document why they diverge.
