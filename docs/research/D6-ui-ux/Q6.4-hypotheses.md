# D6.Q6.4 Hypotheses - Error State Audit

## H6.4.1 - Shared error UI exists but is not the default

Prediction: `ErrorStateWidget` exists and is used in some progress/detail paths, but many async branches still use local `Text(loadErrorLabel)` or raw `error.toString()`.

Falsifier: most feature async error branches use `ErrorStateWidget` or a feature-specific equivalent with consistent semantics.

## H6.4.2 - Retry affordances are uneven

Prediction: some error cards expose retry via `ref.invalidate(...)`, while many blocking error states show only a label.

Falsifier: retry is present on most recoverable async failures.

## H6.4.3 - Optional dashboard errors can disappear

Prediction: home/me optional cards sometimes return `SizedBox.shrink()` on error.

Falsifier: optional panels consistently show compact error or fallback copy.

## H6.4.4 - Raw exception text reaches learner-facing UI

Prediction: several branches render `error.toString()` or append `$err` to copy.

Falsifier: raw exception strings are restricted to logs/debug surfaces.

## H6.4.5 - Snackbars are used for action errors, but not enough for blocking data errors

Prediction: snackbars are common for action failures, while route-level load failures need persistent inline recovery.

Falsifier: blocking load failures are mostly handled with persistent, contextual, retryable UI.
