# D6.Q6.1 Hypotheses - Card Pattern Audit

## H6.1.1 - Shared card primitives exist but are not universal

Prediction: `AppSectionCard`, `AppFeatureCard`, and `AppCompactRow` cover many newer hub surfaces, but home, practice, and test modules still use bespoke `Container`/`Card` surfaces. Effect size: at least 5 of 20 sampled cards are custom.

Falsifier: 18+ sampled cards use the same shared primitive or an explicit wrapper around it.

## H6.1.2 - Border radius values are inconsistent with tokens

Prediction: sampled surfaces use several radius values (`10`, `12`, `16`, `18`, `20`, `22`, `24`, `28`) and the canonical `AppSectionCard` radius `28` does not match `AppSpacing` comments for cards/panels.

Falsifier: sampled modules use only 2-3 documented semantic radii.

## H6.1.3 - Home has a second surface system

Prediction: home widgets use `HomeSurface.softPanel()` with radius `24`, padding `14-16`, and separate shadows instead of always using `AppSectionCard`.

Falsifier: home widgets are thin wrappers around `AppSectionCard`.

## H6.1.4 - Practice modules intentionally diverge

Prediction: learn/grammar/test practice cards use task-state surfaces with state colors, choice tiles, and prompt panels, so forcing them into marketing-style feature cards would reduce clarity.

Falsifier: practice modules are visually inconsistent for accidental reasons and do not encode task state.

## H6.1.5 - CTA placement has no single rule

Prediction: CTAs appear as bottom button wraps, header actions, trailing arrows, full-card taps, and row taps. Effect size: at least 4 CTA patterns in 20 samples.

Falsifier: CTA placement is mostly one convention per screen type.
