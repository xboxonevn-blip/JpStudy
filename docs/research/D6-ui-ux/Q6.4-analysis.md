# D6.Q6.4 Analysis - Error State Audit

## E6.4 Result

Error handling is better than loading handling because strong primitives exist: `ErrorStateWidget`, feature-specific `AppFeatureCard` errors, and localized retry labels. The problem is uneven adoption. Some routes are recoverable and friendly, while others hide errors, show only a generic label, or expose raw exception strings.

## Findings

1. `H6.4.1` supported. `ErrorStateWidget` has `13` feature usages after excluding its own file/test, but broad error/retry search found `269` matches in `69` files. Many branches still use local text or feature-specific error UI.
2. `H6.4.2` supported. `ErrorStateWidget` supports `onRetry`, but the sampled shared-widget usages only showed retry in the home mini dashboard. Strong local patterns like Kanji Hub, Vocab Catalog, Search, Minna Catalog, and Immersion Home do expose retry.
3. `H6.4.3` supported. At least eight sampled error branches collapse with `SizedBox.shrink()`, including home/me optional panels and one Hajimete catalog sub-branch.
4. `H6.4.4` supported. Foundations, Grammar, and Recall Sprint include raw `error.toString()` or `$err` in learner-facing UI.
5. `H6.4.5` supported. Snackbars are appropriate for action failures like import/export/reset, but blocking data errors need persistent inline recovery.

## Production Impact

This is a beta trust issue. Friendly feature-level error cards already show the desired direction, but a learner can still hit raw exception text or a blank optional panel. That makes support harder because users cannot tell whether they should retry, change level, wait, or leave the feature.

The highest-risk cases are practice/session blockers and primary content routes: Recall Sprint, Grammar detail/ghost, Foundations references, and catalog routes. Optional dashboard collapse is lower risk but should still log or show a compact beta hint when data fails.

## Red Team

Steelman opposite: raw error text can help testers report exact failures, and hidden optional-panel errors reduce noise. That is valid in debug tooling, but not learner-facing beta UI. A better compromise is friendly UI plus debug logging/telemetry, not raw exception copy or silent disappearance.

## Decision

No broad one-pass refactor. Define severity rules:

1. Primary route/data errors: persistent visible UI, localized message, no raw exception, retry if the provider can be invalidated.
2. Practice/session errors: same as primary route errors, plus copy that names the blocked session/review bank.
3. Optional dashboard panels: compact inline error or beta/debug telemetry; collapse only for known low-value absence, not unknown failure.
4. Action errors: snackbar is acceptable when the user remains on a valid screen.
5. Raw exception details: logs/debug only, never default learner copy.

## Action

Next implementation should be `Q6.4b`:

1. Convert Foundations and Recall Sprint raw `error.toString()` branches to `ErrorStateWidget` or a feature card with retry.
2. Convert Grammar root/detail/ghost raw `$err` copy to friendly localized error UI.
3. Add `onRetry` to shared-widget usages where provider invalidation is available.
4. Replace optional dashboard error shrink with compact error state or telemetry before beta.
5. Add focused widget tests for one converted primary route and one optional panel.
