# D7.Q7.3 Analysis - Web Renderer Choice

## Verdict

Keep the default `dart2js` + CanvasKit build for the next beta deploy. Do not switch to `--wasm` until live Hosting smoke, browser compatibility, and header policy are proven.

The Wasm build does work locally, but it emits a dual build: primary `dart2wasm` + Skwasm and fallback `dart2js` + CanvasKit. That raises raw deploy output from `62.48 MB` to `68.34 MB` (`+9.4%`). Local smoke looked promising (`21` resources vs `39`, FCP `468ms` vs `540ms`), but the run had security-software injection noise and is not enough to justify a renderer switch.

## Findings

1. Default build is CanvasKit.
   - `compileTarget=dart2js`
   - `renderer=canvaskit`
   - `mainJsPath=main.dart.js`

2. `--wasm` build is dual-path, not a simple replacement.
   - Primary: `compileTarget=dart2wasm`, `renderer=skwasm`, `mainWasmPath=main.dart.wasm`.
   - Fallback: `compileTarget=dart2js`, `renderer=canvaskit`, `mainJsPath=main.dart.js`.
   - `main.dart.wasm` adds `5.79 MB` raw / `2.09 MB` gzip.

3. Artifact size gets worse before live performance is proven.
   - Default total: `62.48 MB`.
   - Wasm total: `68.34 MB`.
   - Delta: `+5.86 MB` raw / `+9.4%`.

4. Local smoke is not decisive.
   - Wasm smoke had fewer resource entries and lower FCP in this run.
   - Both smokes were contaminated by local Kaspersky resource entries.
   - Browser cache, CDN, Firebase headers, and App Check behavior can change the real result.

5. Header policy needs a live decision before Wasm beta.
   - Current CSP includes `'wasm-unsafe-eval'`.
   - Current Hosting headers do not include COOP/COEP.
   - That may be acceptable for the observed local build path, but it is not a launch proof for all Wasm/Skwasm modes.

## Hypothesis Results

- H7.3.1 supported: default build uses `dart2js` + CanvasKit.
- H7.3.2 supported: Wasm build passes and increases raw output by `9.4%`.
- H7.3.3 supported: do not make Wasm the beta default without live compatibility checks.
- H7.3.4 supported: resource-count smoke alone cannot decide renderer choice.

## Product Impact

Renderer choice is now a measurable release-policy decision, not a generic "Flutter web is big" concern.

For beta, use the default renderer path because it is the current known-good build and CI path. Keep Wasm as an experiment behind a separate preview channel once live headers and browser matrix are explicit.
