# D7.Q7.5 Analysis - Asset Compression and Brotli

## Verdict

Brotli is already active on primary Firebase Hosting. Do not add repo-managed precompressed assets for beta.

The current release serves `main.dart.js`, CanvasKit wasm, JSON, and even PNG assets with `Content-Encoding: br` when the client advertises Brotli. The remaining performance work is not "turn compression on"; it is reducing what must be fetched and parsed, measuring live timing, and optimizing assets that compression cannot meaningfully shrink.

## Findings

1. Primary Hosting negotiates Brotli.
   - `main.dart.js` returns `Content-Encoding: br` and `Content-Length: 1,306,529`.
   - `canvaskit/canvaskit.wasm` returns `Content-Encoding: br` and `Content-Length: 2,225,799`.
   - `grammar_n1_1.json` returns `Content-Encoding: br` and `Content-Length: 1,765`.
   - `Vary: x-fh-requested-host, accept-encoding` is present.

2. Brotli is a real win over gzip.
   - `main.dart.js`: `1.31 MB` Brotli vs `1.75 MB` gzip, about `25%` smaller than gzip.
   - `canvaskit.wasm`: `2.23 MB` Brotli vs `2.90 MB` gzip, about `23%` smaller than gzip.
   - Full `build/web`: `62.49 MB` raw, `17.46 MB` gzip, `13.65 MB` Brotli.

3. JSON compresses extremely well, but lazy loading still matters.
   - JSON assets total `19.35 MB` raw but `1.99 MB` Brotli.
   - That lowers transfer cost, but parsing and request fan-out still matter. D7.Q7.1-Q7.4 lazy-loading/resource-count work remains valid.

4. PNG is not a compression problem.
   - PNG assets total `1.06 MB` raw and `1.04 MB` Brotli.
   - `mascot_fox_transparent.png` is `612,019` raw and `606,044` Brotli. If mascot transfer matters, optimize image format/dimensions, not HTTP compression.

5. Cache policy is uniform and conservative.
   - Representative responses use `Cache-Control: max-age=3600`.
   - That is acceptable for beta safety, but a later release task can split cache policy between hashed/static assets and update-sensitive files.

## Hypothesis Results

- H7.5.1 supported: Firebase Hosting serves Brotli for JS, wasm, JSON, HTML, and PNG.
- H7.5.2 supported: Brotli materially improves transfer size versus gzip.
- H7.5.3 supported: PNG sees only about `2%` aggregate savings and needs image optimization if it becomes material.
- H7.5.4 supported: pre-generating `.br`/`.gz` artifacts is unnecessary for current Firebase Hosting.
- H7.5.5 supported: remaining D7 risk is live timing/resource shape, not missing compression.

## Product Impact

For beta, keep the primary-only Hosting flow as-is. Compression is not the blocker.

Next performance work should be:

1. Live route resource smoke against `https://jpstudy.web.app`, not local only.
2. Lighthouse/trace once tool availability is stable.
3. Image audit for mascot and any first-viewport media.
4. Optional cache-header split after launch risk settles.
