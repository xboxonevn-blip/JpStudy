# D7.Q7.4 Analysis - Code Splitting Feasibility

## Verdict

Route-level code splitting is feasible, but should be a controlled post-gate pilot, not a broad refactor during onboarding gating.

The app currently emits one main app JS bundle and zero deferred chunks. That is expected: there are no `deferred as` imports, and route modules eagerly import feature screens. A pilot should start with one heavy leaf route and a reusable eager `DeferredRouteScreen` wrapper.

## Findings

1. Current code has no split boundary.
   - `rg` found no `deferred as` or `loadLibrary()` usage.
   - `build/web` contains no `*.part.js`, `*.deferred*.js`, or `main.dart.js_*` artifacts.
   - `main.dart.js` is `6,278,448` bytes raw in the current release build.

2. Router imports are eager.
   - Route modules import `61` feature libraries across `65` `GoRoute` declarations.
   - The largest eager route modules are home, meta, vocab, exam, and kanji routes.
   - `app_route_builders.dart` imports learn and grammar screens and is reused by six route modules, so it can keep feature code eager even when one route screen is deferred.

3. Split candidates exist.
   - Best first candidates are heavy leaf experiences: JLPT mock pro, immersion reader, handwriting practice, and vocab match session.
   - Avoid first-route and boot-critical code: onboarding, home, shell scaffold, and core route redirect logic.

4. Expected payoff is startup parse/compile, not raw deploy size.
   - D7.Q7.2 showed raw build size is mostly CanvasKit/Skwasm (`52.6%`) and content/support JSON (`30.8%`).
   - App JS is only about `10.6%` of raw build output.
   - Splitting one leaf route will not solve total artifact size, but can keep unused feature JS out of the initial route.

5. Feasibility depends on preserving typed route args.
   - `state.extra` models such as vocab/test args should stay in small eager model files.
   - Deferred screens can still import those models, but route modules should not import the heavy screen library directly.

## Recommended Pilot

Implement one pilot after Phase 6 or later, once navigation gating settles:

```dart
import 'package:jpstudy/features/jlpt/screens/jlpt_mock_pro_screen.dart'
    deferred as jlpt_mock_pro;

GoRoute(
  path: AppRoutePath.jlptMockPro,
  builder: (context, state) => DeferredRouteScreen(
    loadLibrary: jlpt_mock_pro.loadLibrary,
    builder: () => jlpt_mock_pro.JlptMockProScreen(),
  ),
);
```

Pilot gate:

- `flutter analyze lib test`
- focused route tests for direct link and navigation
- `flutter build web --release`
- verify at least one deferred chunk appears
- run `npm run test:web-resource-smoke`
- compare `/` resource count and `main.dart.js` gzip against D7 budget

## Hypothesis Results

- H7.4.1 supported: no deferred imports and no chunks exist.
- H7.4.2 supported: Dart/Flutter support deferred loading, and current route shape can host a wrapper-based pilot.
- H7.4.3 supported: shell/home/onboarding should not be first split targets.
- H7.4.4 supported: raw-size impact is bounded because renderer/content dominate.
- H7.4.5 supported: shared route builders and typed args need cleanup before broad deferral.

## Product Impact

For beta, keep focusing on lazy content fetch, live resource smoke, and onboarding route correctness. Code splitting is a good performance follow-up once P2-P5 route behavior is stable.

Do not start a broad code-splitting refactor until the pilot proves a real chunk, no route regression, and no first-route resource increase.
