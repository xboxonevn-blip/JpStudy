# D7.Q7.2 Analysis - Bundle Composition

## Verdict

The raw release bundle is dominated by renderer/runtime and static content, not only app Dart code.

`build/web` is `62.48 MB` raw. CanvasKit/Skwasm files account for `52.6%`, content/support JSON for `30.8%`, and top-level app JS for `10.6%`. This changes the next optimization order: keep lazy content loading, then evaluate renderer mode/CDN strategy before spending much effort shaving small Dart UI code paths.

## Findings

1. `main.dart.js` is large but not the largest raw family.
   - `main.dart.js`: `6.27 MB` raw.
   - Top-level app JS total: `6.65 MB` raw.
   - CanvasKit/Skwasm family: `32.88 MB` raw.

2. Content/support JSON remains the largest app-owned payload family.
   - `assets/assets/data`: `19.24 MB` raw.
   - Largest JSON files are kanji stroke support and upper-level vocab chapters.
   - D7.Q7.1 lazy seeding still matters because these bytes are static output, while first-route request count determines what the browser actually fetches at startup.

3. No route-level deferred chunks are emitted.
   - No `part`, `deferred`, or `main.dart.js_` chunk files were found.
   - Current build shape is one main app JS bundle plus static renderer/runtime/assets.

4. Source-map inspection is useful but incomplete.
   - `--source-maps` build produced `main.dart.js.map` with `1,464` sources.
   - Source count clusters are Flutter/framework (`426`), app features (`209`), Flutter engine (`142`), Dart SDK (`107`), Drift (`69`), app data/core/app (`97` combined), SQLite (`29`), Riverpod (`29`), and crypto/Firebase families.
   - This is source-file count, not byte attribution. Precise app/package byte ownership needs source-map size tooling.

## Hypothesis Results

- H7.2.1 supported: top-level app JS is only `10.6%` raw build bytes.
- H7.2.2 supported: content/support JSON dwarfs images/fonts/manifests.
- H7.2.3 supported: no deferred route chunks were found.
- H7.2.4 supported: source maps expose source families, but not direct byte totals without extra tooling.

## Product Impact

For SM5 startup performance, the main risk is not simply "Flutter app bundle too big." There are three separate surfaces:

- Renderer/runtime payload policy: CanvasKit/Skwasm raw files dominate the build and should be tested in D7.Q7.3.
- Content fetch policy: app-owned JSON is large, but Q7.1 proved lazy seeding can keep startup requests low.
- App code splitting: everything compiles into one `main.dart.js`; route-level deferral is a future feasibility question, not current behavior.

## Next

D7.Q7.3 should compare renderer choices and deployment serving behavior. D7.Q7.4 should test whether route-level deferred loading is feasible without destabilizing router/provider initialization.
