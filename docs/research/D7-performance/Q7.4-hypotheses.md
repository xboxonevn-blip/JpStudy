# D7.Q7.4 Hypotheses - Code Splitting Feasibility

## Question

Can JpStudy add route-level code splitting on Flutter web without destabilizing router, Riverpod, or onboarding boot state?

## Hypotheses

- H7.4.1: Current release output has no deferred chunks because the app has no Dart `deferred as` imports.
- H7.4.2: Route-level deferral is technically feasible for leaf and heavy non-home routes by wrapping deferred imports in a small eager `DeferredRouteScreen`.
- H7.4.3: Deferring top-level shell branches is higher risk than deferring heavy leaf screens because `StatefulShellRoute.indexedStack` expects branch route builders synchronously.
- H7.4.4: Code splitting will not materially reduce raw deploy size because renderer/runtime and content JSON dominate, but it can reduce cold-route JS parse/compile work.
- H7.4.5: Current route helper imports create eager coupling that must be untangled before broad deferral.

## Prediction

Start with one non-critical heavy leaf route, likely `jlptMockPro`, `immersionReader`, or handwriting practice. A small pilot should produce at least one `part.js` or `main.dart.js_*` style artifact in `build/web` without changing public route paths. If no chunk appears, the deferred import boundary was pierced by eager imports.

## Falsifiers

- Flutter web build emits no extra chunk after a real deferred import boundary.
- GoRouter cannot preserve route state while the deferred route displays a loader.
- A deferred pilot breaks tests for deep links or typed `state.extra` arguments.
- The emitted chunks are too small to justify complexity, or first-route resource count increases enough to hurt SM5.
