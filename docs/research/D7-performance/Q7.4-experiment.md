# D7.Q7.4 Experiment - Code Splitting Feasibility

## Setup

- Date: 2026-05-14
- Flutter: 3.41.5 stable, Dart 3.11.3
- Build inspected: current release `build/web` after primary Hosting deploy
- Source-map build reused from D7.Q7.2: `build/web_q72_sourcemap/main.dart.js.map`

## Method

1. Checked the Flutter/Dart toolchain version.
2. Searched `lib` and `test` for `deferred as`, `loadLibrary`, and generated part files.
3. Inspected `build/web` for deferred chunk artifacts.
4. Counted route module feature imports and `GoRoute` declarations.
5. Counted feature source presence in the Q7.2 source map.
6. Ranked `lib/features/*` by local Dart line count as a rough split-candidate proxy.
7. Reviewed current router shape: eager route modules under `lib/app/navigation/routes/*` plus shared helper imports.

## Commands

```powershell
flutter --version
rg -n "deferred as|loadLibrary\(|Deferred|part '|part of" lib test pubspec.yaml
Get-ChildItem build\web -Filter "*.js" | Select-Object Name,Length
Get-ChildItem build\web -Recurse -Include "*.part.js","*.deferred*.js","main.dart.js_*"
Get-ChildItem lib\app\navigation\routes -Filter *.dart | ...
Get-ChildItem lib\features -Directory | ...
```

## Result

- No app deferred imports exist.
- No deferred chunk artifacts exist in `build/web`.
- Top JS files are `main.dart.js` (`6,278,448` bytes), `drift_worker.js` (`355,298`), `flutter_bootstrap.js` (`9,975`), and `flutter.js` (`9,553`).
- Route modules eagerly import `61` feature libraries across `65` `GoRoute` declarations.
- The heaviest route modules by feature imports are `home_routes.dart` (`11`), `meta_routes.dart` (`10`), `vocab_routes.dart` (`9`), `exam_routes.dart` (`8`), and `kanji_routes.dart` (`6`).
- The source map includes `209` app feature sources. Leading clusters: `home` (`30`), `learn` (`20`), `jlpt` (`17`), `vocab` (`16`), `foundations` (`14`), `test` (`14`), `grammar` (`10`), `write` (`10`).
- Local source line-count proxy ranks heavy feature areas as `home` (`10,722` lines), `vocab` (`8,934`), `jlpt` (`7,813`), `grammar` (`6,946`), `write` (`5,986`), `test` (`5,103`), `lesson` (`5,009`).

## Observations

`app_route_builders.dart` is a coupling hotspot. It is imported by six route modules and eagerly imports `GrammarPracticeScreen`, `LearnScreen`, grammar question generation, and learn models. That means a naive deferred screen import can still leave meaningful feature code in `main.dart.js`.

The safest pilot is a leaf screen that does not participate in shell boot, onboarding redirect, or core home rendering. Good candidates:

1. `JlptMockProScreen`
2. `ImmersionReaderScreen`
3. `HomeHandwritingPracticeScreen` / handwriting practice
4. `VocabMatchSessionScreen`

Avoid deferring these first:

- `HomeScreen`, because it is first route and has the goal banner.
- Onboarding screens, because they gate boot and must stay deterministic.
- `AppShellScaffold`, because it owns persistent nav.
- Shared model/args files, unless moved to a small eager route-args package.

## Surprise

Mild. Code splitting is more feasible than the current one-bundle output suggests, but the first useful split is not "defer a route file." The route helper graph and typed argument models must be shaped first, otherwise deferred boundaries are easy to pierce.
