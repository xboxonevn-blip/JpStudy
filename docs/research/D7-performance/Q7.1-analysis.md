# D7.Q7.1 Analysis - Web Performance Baseline

## E7.1 Result

Release web build passes, but D7 does not yet have a real Lighthouse score. The fallback baseline shows a manageable gzipped app shell (`main.dart.js` `1.77 MB` gzip), a large renderer/runtime footprint, and a bigger product concern: initial load observes `250` resources and eager grammar JSON fetches across levels.

## E7.2 Tooling Update

D7 now has a deterministic build-artifact budget gate:

- `docs/research/D7-performance/web_perf_budget.json`
- `tool/research/web_perf_budget_report.dart`
- `test/tool/research/web_perf_budget_report_test.dart`

Current `build/web` passes the checked-in budgets. This does not replace Lighthouse or live route smoke, but it prevents silent growth in the largest structural artifacts: `main.dart.js`, CanvasKit wasm, sqlite wasm, total build bytes, asset bytes, and JSON bytes.

## Findings

1. `H7.1.1` supported. Release web build passed in `51.9s`.
2. `H7.1.2` supported. `main.dart.js` is `6.25 MB` raw but `1.77 MB` gzip.
3. `H7.1.3` supported. Total assets are `22.17 MB`, JSON assets are `19.35 MB`, and local initial load observed `250` resources with many grammar JSON files.
4. `H7.1.4` supported. Lighthouse CLI is not installed and Chrome/Edge were not on PATH, so no Lighthouse score was produced.
5. `H7.1.5` supported. Localhost release smoke produced Firebase analytics/installations `403` errors due referer restrictions.

## Production Impact

The build is shippable from a compile standpoint, but not performance-budgeted. The app shell size is normal for Flutter web plus Drift/sqlite/CanvasKit, yet first-session performance can still degrade badly if the app eagerly loads broad content banks before the learner needs them.

Highest-risk items:

1. Startup content loading: `250` observed resources and many grammar example JSON files on first local route.
2. Renderer/runtime size: CanvasKit wasm `7.16 MB` raw / `2.92 MB` gzip, plus sqlite wasm `0.74 MB` raw / `0.36 MB` gzip.
3. Asset payload: JSON assets total `19.35 MB`.
4. Local analytics 403 noise blocks clean perf smoke unless Firebase referer config or local analytics disablement is handled.
5. No reproducible Lighthouse command in repo or dev dependencies. E7.2 adds a build-artifact budget fallback, not a Lighthouse score.

## Red Team

Steelman opposite: localhost timing is fast (`FCP 248 ms`) and gzip makes the shell less alarming. Firebase Hosting will serve compression/CDN headers better than Python `http.server`. That lowers confidence in absolute timing, but not in the structural signals: bundle/asset sizes, missing Lighthouse tooling, and broad eager JSON fetches are real enough to set a budget.

## Decision

Do not claim SM5 performance readiness yet. Establish D7 budget and tooling before optimizing:

1. Add a reproducible Lighthouse or Playwright perf script that can run against `build/web` or live hosting.
2. Budget gates now exist for build artifacts and JSON bytes; route-level resource count still needs browser instrumentation.
3. Investigate why initial load fetches grammar examples across many levels.
4. Consider lazy-loading grammar/example banks by selected route and level.
5. Add a local analytics-off mode or localhost referer allowance for clean perf smoke.

## Action

Next implementation should be `Q7.1b`:

1. Extend the checked-in perf command with browser resource-count capture.
2. Add a test or instrumentation guard that catches all-level content prefetch on home/root.
3. Install or document Lighthouse prerequisites, or use Playwright trace metrics as the official fallback.
