# D7.Q7.1 Analysis - Web Performance Baseline

## E7.1 Result

Release web build passes, but D7 does not yet have a real Lighthouse score. The fallback baseline shows a manageable gzipped app shell (`main.dart.js` `1.77 MB` gzip), a large renderer/runtime footprint, and a bigger product concern: initial load observes `250` resources and eager grammar JSON fetches across levels.

## E7.2 Tooling Update

D7 now has a deterministic build-artifact budget gate:

- `docs/research/D7-performance/web_perf_budget.json`
- `tool/research/web_perf_budget_report.dart`
- `test/tool/research/web_perf_budget_report_test.dart`

Current `build/web` passes the checked-in budgets. This does not replace Lighthouse or live route smoke, but it prevents silent growth in the largest structural artifacts: `main.dart.js`, CanvasKit wasm, sqlite wasm, total build bytes, asset bytes, and JSON bytes.

## E7.3 Lazy Grammar Seed Update

The broad first-load grammar fetch had two roots, not one: the content DB was seeding all grammar levels on open, and `databaseProvider` also kicked `GrammarSeeder.seedGrammarData()` on startup, which loaded every level into the app DB. Both are now scoped by learner level with on-demand fallbacks for non-active routes.

The fix keeps first startup level-scoped while protecting advanced paths:

- startup app grammar seed reads `onboarding.level` and seeds that level only;
- grammar repository requests seed their requested level when missing;
- content DB exposes `ensureGrammarSeededForLevel(level)`;
- lesson grammar sync and JLPT mock sections call the content-level ensure before direct `grammarPoint` queries;
- app lesson grammar queries now filter by `lessonId + jlptLevel`, avoiding N1/N2/N3 lesson-id collisions with N5 lesson `1`.

## E7.4 Browser Re-Measure

After rebuilding `build/web`, local Playwright smoke dropped first-route resources from `250` to `108` (`-56.8%`). The grammar fetch is no longer broad/all-level: the observed grammar resources are `50` N5-only grammar definition/example files. This supports the E7.3 root-cause hypothesis.

Residual issue: the app still fetches a full active-level grammar bank (`25` definition + `25` example files) on first load. That is much safer than N1-N5, but still not route-minimal. A stricter budget should now target active-level grammar prefetch and move grammar seeding closer to the grammar route or first lesson usage.

## E7.5 Startup Seeder Ablation

Removing the fire-and-forget app DB grammar seed from `databaseProvider` dropped first-route resources from `108` to `69`; grammar JSON dropped from `50` to `0`, and only `manifest.json` plus `FontManifest.json` remained in the JSON sample. On-demand grammar seed still passes through `GrammarRepository.fetchPointsByLevel`, and focused grammar tests stayed green.

This is the best D7 result so far: startup route now avoids grammar content entirely. The next performance question is not "why is root loading grammar?" but "can we automate this resource-count gate and then inspect the remaining non-content requests, especially Firebase localhost noise and Drift/runtime assets?"

## E7.6 Resource Smoke Gate

D7 now has a checked-in Playwright resource-count gate:

```text
npm run test:web-resource-smoke
```

The gate serves `build/web`, launches Chromium, waits for first route settle, reads PerformanceResourceTiming, and enforces:

- `maxResources = 80`
- `maxJson = 5`
- `maxGrammar = 0`

Local run passed at `resourceCount=38`, `jsonCount=1`, `grammarResourceCount=0`. CI now installs Node dependencies, installs Playwright Chromium, runs the unit test, and runs the smoke after the release web build and artifact budget.

## Findings

1. `H7.1.1` supported. Release web build passed in `51.9s`.
2. `H7.1.2` supported. `main.dart.js` is `6.25 MB` raw but `1.77 MB` gzip.
3. `H7.1.3` supported and materially remediated. Total assets are `22.17 MB`, JSON assets are `19.35 MB`, but local initial load dropped from `250` to `69` resources after level-scoping grammar seeders and removing app DB startup grammar seeding. First-route grammar JSON is now `0`.
4. `H7.1.4` supported. Lighthouse CLI is not installed and Chrome/Edge were not on PATH, so no Lighthouse score was produced.
5. `H7.1.5` supported. Localhost release smoke produced Firebase analytics/installations `403` errors due referer restrictions.

## Production Impact

The build is shippable from a compile standpoint, but not performance-budgeted. The app shell size is normal for Flutter web plus Drift/sqlite/CanvasKit, yet first-session performance can still degrade badly if the app eagerly loads broad content banks before the learner needs them.

Highest-risk items:

1. Startup content loading: post-fix local route is `69` resources, down from `250`, with `0` grammar JSON resources.
2. Renderer/runtime size: CanvasKit wasm `7.16 MB` raw / `2.92 MB` gzip, plus sqlite wasm `0.74 MB` raw / `0.36 MB` gzip.
3. Asset payload: JSON assets total `19.35 MB`.
4. Local analytics 403 noise blocks clean perf smoke unless Firebase referer config or local analytics disablement is handled.
5. No reproducible Lighthouse command in repo or dev dependencies. E7.2 adds a build-artifact budget fallback, not a Lighthouse score.

## Red Team

Steelman opposite: localhost timing is fast (`FCP 248 ms`) and gzip makes the shell less alarming. Firebase Hosting will serve compression/CDN headers better than Python `http.server`. That lowers confidence in absolute timing, but not in the structural signals: bundle/asset sizes, missing Lighthouse tooling, and broad eager JSON fetches are real enough to set a budget.

## Decision

Do not claim SM5 performance readiness yet. Establish D7 budget and tooling before optimizing:

1. Add a reproducible Lighthouse or Playwright perf script that can run against `build/web` or live hosting.
2. Budget gates now exist for build artifacts, JSON bytes, and first-route resource count.
3. Lazy-load grammar/example banks by selected route and level is implemented for the discovered app/content DB seeding paths; app startup no longer seeds grammar.
4. Add a local analytics-off mode or localhost referer allowance for clean perf smoke.
5. Next D7 work should move from root resource count to live Firebase Hosting verification, Lighthouse, and remaining large font/runtime assets.

## Action

Next implementation should be `Q7.1b`:

1. Run the resource-count gate on live Firebase Hosting after deploy, not only local `build/web`.
2. Install or document Lighthouse prerequisites, or use Playwright trace metrics as the official fallback.
3. Audit remaining large first-route resources: `main.dart.js`, Drift worker, Noto Color Emoji, and remote font requests.
