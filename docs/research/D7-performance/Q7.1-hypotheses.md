# D7.Q7.1 Hypotheses - Web Performance Baseline

## H7.1.1 - Release web build passes

Prediction: `flutter build web --release --base-href=/` succeeds from the current workspace.

Falsifier: release web build fails or requires missing secrets.

## H7.1.2 - Main JS is large but gzip-manageable

Prediction: Flutter web produces a multi-megabyte `main.dart.js`, but gzip size stays near or under `2 MB`.

Falsifier: `main.dart.js` gzip size exceeds `3 MB` before content assets.

## H7.1.3 - Content assets are the larger launch risk

Prediction: JSON/content assets and startup fetch shape are more important than the app shell bundle alone.

Falsifier: initial release load fetches only a small fixed route-specific asset set.

## H7.1.4 - Lighthouse score cannot be trusted until local tooling exists

Prediction: Lighthouse is not available locally, so D7 must record build-size and Playwright timing as a fallback baseline.

Falsifier: Lighthouse CLI and a browser binary are available and can produce a report.

## H7.1.5 - Localhost Firebase restrictions add audit noise

Prediction: local release serving triggers Firebase referer restrictions that affect console/network logs but are not the same as live-hosting failures.

Falsifier: local run has no Firebase 403s or analytics/installations errors.
