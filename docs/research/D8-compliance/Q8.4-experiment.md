# D8.Q8.4 Experiment - Error Monitoring Design

## Experiment E8.8

Question: Is production error monitoring already wired, and which monitoring path should be chosen for web beta?

Method:

- Search source, tests, docs, and workflow files for Crashlytics, Sentry, global error hooks, crash handlers, non-fatal recorders, and source-map upload.
- Read `pubspec.yaml`, `lib/main.dart`, `lib/core/analytics/*`, `SHIPPING.md`, `docs/FIREBASE_SECURITY_CHECKLIST.md`, and `.github/workflows/ui-string-guard.yml`.
- Compare the repository state with current official Firebase Crashlytics for Flutter docs and Sentry Flutter docs.
- Classify a minimum beta gate and a deferred implementation path.

Commands:

```text
rg "Crashlytics|crashlytics|Sentry|sentry|FirebaseCrashlytics|monitor|error reporting|analytics" lib pubspec.yaml firebase.json .github docs -n
rg "FlutterError.onError|PlatformDispatcher|runZonedGuarded|ErrorWidget|crash|nonfatal|recordError|FirebaseCrashlytics|SentryFlutter|ErrorReporter|logError" lib test -n
Get-Content pubspec.yaml
Get-Content lib\main.dart
Get-Content lib\core\analytics\analytics_service.dart
Get-Content .github\workflows\ui-string-guard.yml
```

External sources:

- Firebase Crashlytics Flutter setup: https://firebase.google.com/docs/crashlytics/flutter/get-started
- Sentry Flutter setup: https://docs.sentry.io/platforms/flutter/
- Sentry Flutter troubleshooting: https://docs.sentry.io/platforms/flutter/troubleshooting/

Expected information gain: medium-high. This distinguishes "dataset exists" from "client captures crashes" and forces a web-vs-mobile monitoring decision before beta.

Limitations:

- This is a design/audit question, not a vendor procurement decision.
- Firebase Console Crashlytics state and Sentry org/project state are not inspectable from repository files.
- The final vendor choice still needs privacy/legal review because error events can contain device, URL, stack, release, and user context.
