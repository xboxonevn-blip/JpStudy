# D8.Q8.6 Hypotheses - Release Process Integration

## Question

Is the release process now integrated enough to prevent stale-target and legacy-hosting regressions during the onboarding-gate rollout?

## Hypotheses

- H8.6.1: Active release docs can be made primary-only now that the deploy target is chosen.
- H8.6.2: `firebase deploy --only hosting:jpstudy` refreshes primary Hosting without re-enabling the disabled default legacy site.
- H8.6.3: A post-deploy check must include both primary `200` and legacy `404`/`SITE_DISABLE`, because a stray local target can otherwise re-enable the old host.
- H8.6.4: Full CI deploy automation should remain deferred until Firebase credentials, App Check site key, live smoke, and notification policy are explicit.
- H8.6.5: `firebase deploy --only hosting` is unsafe for this repo because it can deploy multiple hosting targets if local target drift returns.

## Prediction

After a primary-only deploy, `https://jpstudy.web.app` should return `200` with security headers and `https://jpstudy-v2.web.app` should return `404`. Firebase channel metadata should show primary release type `DEPLOY` and legacy release type `SITE_DISABLE`.

## Falsifiers

- A primary-only deploy reactivates `jpstudy-v2`.
- Active docs still instruct generic all-hosting deploy.
- CI deploy automation exists without a live smoke or notification gate.
- App Check build flag is missing from the release command.
