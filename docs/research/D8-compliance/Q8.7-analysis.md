# D8.Q8.7 Analysis - Telemetry Retention and GDPR

## Verdict

Telemetry is acceptable for a small closed beta only if the beta notes explicitly call out the remaining deletion/retention gaps. It is not public-launch ready.

The app has a strong starting point: Analytics is consent-gated, Do Not Track disables collection, custom events are coarse and non-free-text, and Data controls now expose a user-triggered device Analytics reset. A beta operations runbook now exists at `docs/compliance/user-data-deletion-runbook.md`. BigQuery export now has a source-verifiable 60-day expiration policy. Firebase Storage setup/migration proof is beta-deferred by the 2026-05-17 owner decision because the project remains on Spark and beta backup is local file export/import. The remaining operational compliance gaps are first executed deletion proof, source-verifiable GA4 UI retention setting, and a web-specific deletion caveat because the current Firebase Analytics web SDK does not support `resetAnalyticsData()`.

## Findings

1. Consent gate is implemented.
   - `analytics.consent` defaults to absent/false.
   - Do Not Track suppresses the banner and disables collection.
   - `setAnalyticsCollectionEnabled` is called with `isGranted && !doNotTrack`.

2. Custom telemetry is minimized.
   - Event params are level, goal, counts, mode, trigger, rating, and interval.
   - No prompt text, answer text, display name, email, or arbitrary learner text is logged by `AnalyticsService`.
   - The privacy draft says telemetry should not contain prompts, answers, names, or free-text learner content.

3. User data controls are partially complete for telemetry.
   - Data controls keep local file export/import visible for beta backup.
   - Firebase Storage backup deletion is not a beta target because cloud backup and legacy migration are disabled and no Storage data is created.
   - Data controls now include "Reset analytics on this device", backed by `AnalyticsService.resetAnalyticsData()`.
   - The wrapper calls Firebase only after explicit confirmation and reports `unsupported` separately from generic failures.
   - On Flutter web, `firebase_analytics_web 0.6.1+5` still throws `UnimplementedError('resetAnalyticsData() is not supported on Web.')`; this keeps the UI honest but does not erase already received GA4 reports.
   - The support deletion runbook now covers GA Admin API deletion for `properties/536663906`.

4. GA4 retention is a Console/Admin setting, not code.
   - Official GA4 standard retention choices are `2 months` or `14 months` for user-level data and key events.
   - The repo cannot prove which option is configured.
   - For beta, choose and document the setting explicitly. `2 months` is privacy-minimal and still covers D1/D7/D30; `14 months` supports longer cohort analysis but needs a stronger stated reason.

5. BigQuery export retention is proven for the current dataset.
   - `analytics_536663906` exists in `asia-southeast1`.
   - Dataset options show `default_table_expiration_days=60.0` and `default_partition_expiration_days=60.0`.
   - `events_20260514` has `expiration_timestamp` `2026-07-14T02:56:46.272Z`.

6. User deletion targeting is not ready.
   - E8.11 now calls `setUserId` and `auth_type` only through the consent-gated `AnalyticsService`, after an existing user is present or anonymous sign-in succeeds.
   - GA deletion can be submitted by app instance ID, client ID, user ID, or normalized user-provided data. The app does not currently expose/capture those identifiers for support.
   - The support runbook now defines the beta deletion sequence across Firebase Auth, GA4 user deletion, and BigQuery export cleanup. Firebase Storage deletion is future-only unless Storage is explicitly reintroduced.
   - Data controls now exposes a support ID copy action for signed-in users, including anonymous Firebase users, so support can target UID-based deletion.

7. Anonymous Auth bootstrap now provides a zero-friction UID.
   - On boot, the app attempts Firebase anonymous sign-in with a 5-second timeout after Firebase/App Check init and before `runApp`.
   - Existing signed-in users are reused; anonymous sign-in failure falls back to local-only boot.
   - Legacy progress-like `SharedPreferences` keys are uploaded once to `users/{uid}/legacy_migration.json` and the logical preference `auth.migrated` is set. On web this persists as `flutter.auth.migrated`.
   - Storage rules now allow only the owning UID to read/write/delete that migration file.

## E8.11 Implementation Update

Phase 13 anonymous auth foundation is source-wired and locally tested:

- `AnonymousAuthService` handles existing-user reuse, anonymous sign-in, timeout fallback, Analytics identity, and one-time legacy migration.
- `LegacyMigrationService` snapshots progress-like local preferences, excludes consent state, uploads to Firebase Storage, and only marks migrated after a successful upload.
- `AuthUser` now records `isAnonymous`.
- `main.dart` runs anonymous auth bootstrap before `runApp`.
- Storage rules and emulator tests cover `users/{uid}/legacy_migration.json`.

## 2026-05-15 Live Auth Verification Update

Live App Check bootstrap reaches Firebase Auth. Initial browser network and
REST probes against `https://jpstudy.web.app` returned Identity Toolkit
`ADMIN_ONLY_OPERATION` for `accounts:signUp`, proving Firebase Anonymous Auth
was disabled server-side. The provider was then enabled through the Identity
Toolkit project config API using the logged-in Firebase CLI account. A REST
probe with the production referrer created and deleted one anonymous test user
successfully, and a live browser probe saw `accounts:signUp` return `200`.

Storage update: live legacy migration upload was intentionally kept gated after
Firebase Storage setup was found to require Blaze for new buckets. The
2026-05-17 owner decision descopes Firebase Storage from beta, keeps
`JPSTUDY_ENABLE_LEGACY_STORAGE_MIGRATION` unset/false, and makes local file
export/import the beta backup path. Storage rules, CORS, and service scaffolding
remain checked in for a future cloud-sync rebuild, but Storage provisioning is
not a beta proof gap.

## 2026-05-15 Analytics Reset Update

Data controls now include an Analytics section with a confirmation flow for
resetting local Analytics data on the current device. `AnalyticsService` exposes
`resetAnalyticsData()` and intentionally does not require Analytics collection
to be enabled, because reset is a user privacy action rather than telemetry
emission.

Local test evidence:

- `test/core/analytics/analytics_service_test.dart` verifies reset is called
  even when event collection is disabled.
- The same test verifies web-style `UnimplementedError` returns
  `AnalyticsResetResult.unsupported`.
- `test/features/me/data_settings_screen_test.dart` verifies the Data controls
  button, confirmation dialog, service call, and success snackbar.

Remaining proof gap: this is a device-side SDK control only. It does not satisfy
server-side deletion for GA4 data already received by Google, and web remains
SDK-unsupported until Firebase Analytics web adds `resetAnalyticsData()` or the
app implements a separately reviewed web deletion mechanism.

## 2026-05-15 BigQuery Retention Update

`analytics_536663906` is now present in `asia-southeast1`. The first observed
table is `events_20260514`. Metadata queried via
`tool/research/bigquery_rest_runner.js` shows:

- dataset `default_table_expiration_days`: `60.0`;
- dataset `default_partition_expiration_days`: `60.0`;
- `events_20260514` expiration: `2026-07-14T02:56:46.272Z`.

This clears the BigQuery TTL proof gap for beta operations. It does not clear
the GA4 UI retention proof gap or the need to execute one deletion runbook end
to end.

## Required Before Public Launch

1. Decide and record GA4 retention setting (`2 months` vs `14 months`) with rationale.
2. Execute and evidence one support-runbook deletion across Firebase Auth, GA4 user deletion, and BigQuery export rows. Include Firebase Storage only if cloud backup is explicitly reintroduced before launch.
3. Keep the BigQuery 60-day TTL policy documented and verify future GA4 tables inherit expiration.
4. Update Privacy Policy copy with concrete retention and deletion wording.
5. Track the web Analytics reset limitation in launch notes until a reviewed server-side deletion path covers web users.

## Beta Decision

Closed beta can continue with consent-gated telemetry because the custom event surface is minimized and users can decline. Do not market this as GDPR-complete until deletion and retention operations are tested end to end.
