# D8.Q8.7 Analysis - Telemetry Retention and GDPR

## Verdict

Telemetry is acceptable for a small closed beta only if the beta notes explicitly call out the remaining deletion/retention gaps. It is not public-launch ready.

The app has a strong starting point: Analytics is consent-gated, Do Not Track disables collection, and custom events are coarse and non-free-text. The missing part is operational compliance: no in-app Analytics reset, no GA user-deletion runbook, no source-verifiable GA4 retention setting, and no BigQuery dataset/table expiration proof.

## Findings

1. Consent gate is implemented.
   - `analytics.consent` defaults to absent/false.
   - Do Not Track suppresses the banner and disables collection.
   - `setAnalyticsCollectionEnabled` is called with `isGranted && !doNotTrack`.

2. Custom telemetry is minimized.
   - Event params are level, goal, counts, mode, trigger, rating, and interval.
   - No prompt text, answer text, display name, email, or arbitrary learner text is logged by `AnalyticsService`.
   - The privacy draft says telemetry should not contain prompts, answers, names, or free-text learner content.

3. User data controls are incomplete for telemetry.
   - Data controls can delete Firebase Storage backup.
   - There is no app UI for `FirebaseAnalytics.resetAnalyticsData()`.
   - There is no GA Admin API deletion runbook for `properties/536663906`.

4. GA4 retention is a Console/Admin setting, not code.
   - Official GA4 standard retention choices are `2 months` or `14 months` for user-level data and key events.
   - The repo cannot prove which option is configured.
   - For beta, choose and document the setting explicitly. `2 months` is privacy-minimal and still covers D1/D7/D30; `14 months` supports longer cohort analysis but needs a stronger stated reason.

5. BigQuery export retention is unproven.
   - The GA4 export dataset may not exist yet.
   - The local metadata query could not run because `google-cloud-bigquery` is missing in the active Python.
   - Once `analytics_536663906` exists, check dataset/table expiration and add a TTL policy or scheduled cleanup if raw exports should not persist indefinitely.

6. User deletion targeting is not ready.
   - E8.11 now calls `setUserId` and `auth_type` only through the consent-gated `AnalyticsService`, after an existing user is present or anonymous sign-in succeeds.
   - GA deletion can be submitted by app instance ID, client ID, user ID, or normalized user-provided data. The app does not currently expose/capture those identifiers for support.

7. Anonymous Auth bootstrap now provides a zero-friction UID.
   - On boot, the app attempts Firebase anonymous sign-in with a 5-second timeout after Firebase/App Check init and before `runApp`.
   - Existing signed-in users are reused; anonymous sign-in failure falls back to local-only boot.
   - Legacy progress-like `SharedPreferences` keys are uploaded once to `users/{uid}/legacy_migration.json` and the logical preference `auth.migrated` is set. On web this persists as `flutter.auth.migrated`.
   - Storage rules now allow only the owning UID to read/write/delete that migration file.

## E8.11 Implementation Update

Phase 13 anonymous auth foundation is source-wired and locally tested:

- `AnonymousAuthService` handles existing-user reuse, anonymous sign-in, timeout fallback, Analytics identity, and one-time legacy migration.
- `LegacyMigrationService` snapshots progress-like local preferences, excludes consent state, uploads to Firebase Storage, and only marks migrated after a successful upload.
- `AuthUser` now records `isAnonymous`.
- `main.dart` runs anonymous auth bootstrap before `runApp`.
- Storage rules and emulator tests cover `users/{uid}/legacy_migration.json`.

Remaining proof gap: live App Check + anonymous Auth sign-in should be verified
after the next secret-backed deploy.

## Required Before Public Launch

1. Add an in-app Data control to reset Analytics data on this device.
2. Decide and record GA4 retention setting (`2 months` vs `14 months`) with rationale.
3. Add a support runbook for user deletion across Firebase Auth, Firebase Storage backup, GA4 user deletion, and BigQuery export rows.
4. After GA4 BigQuery export appears, verify or set dataset/table expiration.
5. Update Privacy Policy copy with concrete retention and deletion wording.

## Beta Decision

Closed beta can continue with consent-gated telemetry because the custom event surface is minimized and users can decline. Do not market this as GDPR-complete until deletion and retention operations are tested end to end.
