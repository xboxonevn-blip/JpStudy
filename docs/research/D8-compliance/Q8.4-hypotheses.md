# D8.Q8.4 Hypotheses - Error Monitoring Design

Question: What error monitoring path is safe enough for the first web beta, and what is missing from the repository today?

Source commit: `26138b4c`

## Hypotheses

### H8.4.1 - Firebase Crashlytics is already wired

Prediction: `pubspec.yaml` includes `firebase_crashlytics`, `main.dart` installs `FlutterError.onError` and `PlatformDispatcher.instance.onError`, and the repo has a first-crash verification note.

Expected effect size: beta crash visibility is mostly a console/deploy verification task.

### H8.4.2 - Analytics events are being used as an error-monitoring proxy

Prediction: `AnalyticsService` records exception/error events or failed route states with enough stack/context to triage beta defects.

Expected effect size: no new vendor is required for early beta, but crash triage quality is weaker.

### H8.4.3 - No runtime error sink exists

Prediction: the app has Firebase Analytics consent and product events, but no crash/error SDK, no global Flutter/platform error hooks, and no source-map/debug-symbol upload path.

Expected effect size: beta defects will be found through user reports, local reproduction, and logs, not production error telemetry.

### H8.4.4 - Web beta needs a cross-platform error sink before Crashlytics-only work

Prediction: official Firebase Flutter Crashlytics setup is mobile-oriented, while Sentry's Flutter SDK explicitly supports automatic error monitoring and web-specific setup concerns such as zones and source maps.

Expected effect size: if the beta surface is Firebase Hosting web, a Sentry-style web-capable sink gives higher launch value than mobile-only Crashlytics wiring.
