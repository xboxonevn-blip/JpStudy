# D8.Q8.5 Analysis - CI/CD Gates

## E8.7 Result

CI is stronger than the prompt implied, but not launch-complete. The existing workflow runs PR/push Flutter string guard, `flutter analyze`, `flutter test`, `flutter build web`, and Firebase Storage rules tests.

Missing launch gates:

- post-deploy live route smoke;
- Lighthouse/performance budget on deployed URL;
- explicit failure notification beyond GitHub Checks;
- branch protection proof.

E8.7 remediation changed the existing workflow name to `CI`, made the web build release-like with `--release --base-href=/ --dart-define=JPSTUDY_RECAPTCHA_SITE_KEY=ci-placeholder`, added the D7 build-artifact budget report after the web build, and added the D7 Playwright resource-count smoke gate. This verifies the release build path and local first-route request budget without requiring the real App Check site key in PR CI.

The same build command passed locally, and `npm run test:web-resource-smoke` passed with `resourceCount=39`, `jsonCount=1`, `grammarResourceCount=0`.

## Hypothesis Update

1. `H8.5.1` falsified. CI is not mostly absent.
2. `H8.5.2` initially supported, then partially remediated. CI now uses release/base/App Check define flags with a placeholder key.
3. `H8.5.3` partially supported. Local post-build resource smoke now exists, but no post-deploy live URL smoke or Lighthouse workflow was found.
4. `H8.5.4` supported. Storage rules are covered by a Node job.
5. `H8.5.5` supported from repo evidence. No explicit notification integration was found.

## Beta Impact

PR regressions are better covered. Deploy regressions remain under-tested: a stale or wrong Hosting target can still pass CI, and D7 performance is currently local-build only rather than live-host measured. This matches the D4/D8 pattern: local source can improve while live URLs stay stale.

## Red Team

Steelman opposite: keeping deploy, Lighthouse, and notification out of CI may be intentional while the product is pre-beta and Firebase credentials are not in GitHub Secrets. That is reasonable for now. But before 100-user beta, live-channel proof and performance budgets must be automated or at least encoded as a release gate.

## Decision

Do not add a deploy workflow until the target and secrets policy are explicit. The smallest safe CI improvements have been applied: the existing web build is release-like without requiring real secrets, artifact budget runs after build, and local first-route resource smoke runs after build. Add Lighthouse/live smoke only after the deploy target decision.

Recommended next CI steps:

1. Add Lighthouse CI after live target is chosen.
2. Add post-deploy route smoke after deploy automation exists.
3. Add explicit failure notification or enforce branch protection once GitHub settings are available.

## E8.10 Implementation Update

The live target and branch policy are now explicit: push directly to `main` and
deploy only `hosting:jpstudy` at `https://jpstudy.web.app`. The workflow now has
a `deploy-hosting` job after source CI and Storage rules:

- runs only for `push` on `main`;
- requires `FIREBASE_TOKEN` and `JPSTUDY_RECAPTCHA_SITE_KEY` repository secrets;
- builds with App Check, optional Sentry DSN, and `JPSTUDY_RELEASE`;
- deploys with `firebase deploy --only hosting:jpstudy --project jpstudy-v2`;
- verifies primary `jpstudy.web.app` returns `200`;
- verifies disabled legacy `jpstudy-v2.web.app` returns `404`;
- runs live web resource smoke;
- runs a Lighthouse live gate.

The job intentionally skips deploy with a warning when required secrets are not
set, so normal CI still protects `main` while credential setup is pending.
Remaining proof gaps: first successful secret-backed deploy run, branch
protection, and explicit notification policy.
