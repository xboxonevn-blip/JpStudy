# D8.Q8.3 Analysis - Firebase Auth Authorized Domains

## E8.5 Result

Firebase Hosting origins are known, but the Firebase Auth authorized-domain allowlist is not verified from repository source or the read-only Hosting CLI output.

Confirmed production web origins:

- `https://jpstudy-v2.web.app`
- `https://jpstudy.web.app`

Confirmed Auth config domain in app source:

- `jpstudy-v2.firebaseapp.com`

The pre-existing security checklist only said to review authorized domains/OAuth clients. It did not explicitly require removing `localhost` from the production Firebase Auth authorized-domain list or list the production domains that must remain allowed.

## Hypothesis Update

1. `H8.3.1` supported. Source and Hosting CLI prove project/sites, not Auth authorized-domain Console state.
2. `H8.3.2` supported. The checklist had a generic review line but no production allowlist or `localhost` removal gate.
3. `H8.3.3` supported. Both `jpstudy-v2.web.app` and `jpstudy.web.app` are active Hosting sites and must be considered.
4. `H8.3.4` remains unverified. `localhost` presence in the production Auth allowlist cannot be proven or falsified from current source evidence.

## Beta Impact

This is a manual launch gate, not a code blocker. Before public beta, inspect Firebase Console Auth authorized domains and OAuth client origins. Remove `localhost` from the production project unless a documented dev/beta exception exists. Prefer a separate dev Firebase project for local Auth testing.

## Red Team

Steelman opposite: Firebase may require default project domains such as `jpstudy-v2.firebaseapp.com`, and local development may need `localhost`. That is acceptable only if it is explicit and environment-scoped. The risk is not that `localhost` is always bad; the risk is silent production-project drift.

## Decision

Update `docs/FIREBASE_SECURITY_CHECKLIST.md` with a concrete Auth authorized-domain gate. Mark Q8.3 as measured but console-blocked for final verification.
