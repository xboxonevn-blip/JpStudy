# D8.Q8.2 Hypotheses - Firebase API Key Restriction

Question: Is the Firebase web API key restricted so unauthorized HTTP referrers cannot use Identity Toolkit/Auth APIs?

Commit: `aa5a2c441301bc43f7770325b41f0f4a7dd02bd7`

## Hypotheses

### H8.2.1 - Web API key has HTTP referrer restrictions

Prediction: An Identity Toolkit request with fake `Origin` / `Referer` returns `403` with an API-key referrer restriction error.

Expected effect size: Unauthorized browser-origin abuse surface is materially reduced for the web key.

### H8.2.2 - Web API key has API/service restrictions but no HTTP referrer restriction

Prediction: The same fake-referrer Identity Toolkit request reaches endpoint validation and returns an Auth/API semantic error such as `400`, not `403`.

Expected effect size: The key may be limited by API, quota, App Check, or Auth settings, but referrer-based key restriction is not confirmed.

### H8.2.3 - Web API key is broadly unrestricted

Prediction: Fake-referrer requests to Identity Toolkit are accepted up to normal endpoint validation, and repository docs do not define a restriction/rotation gate.

Expected effect size: Launch risk stays high until GCP Console HTTP referrer restrictions are added and re-tested.

### H8.2.4 - Local docs confuse App Check with API key restriction

Prediction: `docs/FIREBASE_SECURITY_CHECKLIST.md` documents App Check/Auth hardening, but does not explicitly require GCP API-key HTTP referrer restrictions for `jpstudy-v2.web.app` / `jpstudy.web.app`.

Expected effect size: Even if App Check is planned, the manual API-key restriction task can be missed during deploy prep.
