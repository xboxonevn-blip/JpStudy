# D8.Q8.3 Hypotheses - Firebase Auth Authorized Domains

Question: Is Firebase Auth domain authorization launch-ready for production web sign-in, especially around removing `localhost` from production?

Commit: `aa5a2c441301bc43f7770325b41f0f4a7dd02bd7`

## Hypotheses

### H8.3.1 - Authorized-domain list is console-only and not verifiable from source

Prediction: Source/config exposes `authDomain` and Hosting sites, but not the Firebase Auth Console authorized-domain allowlist.

Expected effect size: Launch readiness needs a manual console gate; source-only audit cannot prove `localhost` removal.

### H8.3.2 - Security docs mention authorized domains but do not define a production allowlist

Prediction: `docs/FIREBASE_SECURITY_CHECKLIST.md` says to review authorized domains/OAuth clients, but does not list required production domains or explicitly say to remove `localhost`.

Expected effect size: A deployer can miss the manual console task even if they follow repository docs.

### H8.3.3 - Production web sign-in needs both Firebase Hosting sites considered

Prediction: Firebase Hosting has both `jpstudy-v2.web.app` and `jpstudy.web.app`, while `lib/firebase_options.dart` uses `authDomain: jpstudy-v2.firebaseapp.com`.

Expected effect size: The launch gate needs to confirm all actually deployed origins that users can reach, not just the default Firebase project domain.

### H8.3.4 - Keeping `localhost` in production is acceptable only for an explicit dev project, not the public beta project

Prediction: If `localhost` remains authorized in the production project, local/dev origins can exercise production Auth flows, making sign-in abuse/debug noise harder to separate.

Expected effect size: Moderate security/process risk; fix is manual Console cleanup or separate dev Firebase project.
