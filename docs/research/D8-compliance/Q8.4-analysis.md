# D8.Q8.4 Analysis - Error Monitoring Design

## E8.8 Result

Production error monitoring is not wired. The repository has consent-gated Firebase Analytics and product telemetry, but no `firebase_crashlytics`, no `sentry_flutter`, no global Flutter/framework error hook, no platform async error hook, no source-map upload, and no first-crash verification step.

The visible `firebase_crashlytics` BigQuery dataset is not enough evidence. It currently has zero tables and the app client does not include the Crashlytics package or handlers.

## Hypothesis Update

1. `H8.4.1` falsified. Crashlytics is not wired in the app.
2. `H8.4.2` falsified as an error-monitoring solution. Analytics logs study/auth/sync/product events, but not exceptions, stacks, or fatal/non-fatal errors.
3. `H8.4.3` supported. Runtime error telemetry is absent from source and CI.
4. `H8.4.4` supported for web beta. Firebase Crashlytics remains useful for Android/iOS later, but the immediate deployed beta surface is Firebase Hosting web. A Sentry-style Flutter/web-capable sink is the better first design if one vendor is allowed.

## Beta Impact

Do not claim beta operational readiness until at least one runtime error sink exists and has a verified test event from the deployed channel. Without it, D4 persona retests can prove visible route behavior, but production regressions after deploy will depend on manual reports.

Minimum web-beta monitoring gate:

1. Choose Sentry for Flutter web error monitoring, or explicitly accept "no automated web crash reports" as a launch risk.
2. Add `sentry_flutter` and initialize before `runApp` using `SENTRY_DSN`, `SENTRY_ENVIRONMENT`, and `SENTRY_RELEASE` dart-defines.
3. Keep performance tracing/session replay off by default until privacy/legal review.
4. Build web with source maps for release verification and upload them through CI or a documented release command.
5. Force one non-production test exception and record the issue URL/screenshot in release notes.

Mobile follow-up:

- If Android/iOS beta becomes active, add `firebase_crashlytics` and install the official Flutter framework and platform async handlers.
- Add Crashlytics debug-symbol upload if obfuscation or split debug info is used.

## Red Team

Steelman opposite: adding Sentry introduces a new vendor, DSN/secrets management, privacy review, and bundle/runtime overhead. For a tiny closed beta, manual reports plus Firebase Analytics could be acceptable for a few days. That argument is valid only if the release checklist explicitly records the risk and the beta cohort has a fast feedback channel.

## Decision

Record D8.Q8.4 as measured and blocked on implementation. Recommended path: Sentry for web beta first; Crashlytics for mobile later if native distribution begins. Do not wire either in this docs-only research commit because vendor/secrets/privacy approval is a separate launch decision.
