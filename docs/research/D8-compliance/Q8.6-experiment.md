# D8.Q8.6 Experiment - Release Process Integration

## Setup

- Date: 2026-05-14
- Target project: `jpstudy-v2`
- Active Hosting site: `jpstudy`
- Disabled legacy default site: `jpstudy-v2`

## Method

1. Read active release docs: `SHIPPING.md`, `docs/FIREBASE_SECURITY_CHECKLIST.md`, `.firebaserc`, and `firebase.json`.
2. Deployed Phase 5 with the primary-only command:

```powershell
flutter build web --release --base-href=/ --dart-define=JPSTUDY_RECAPTCHA_SITE_KEY=$env:JPSTUDY_RECAPTCHA_SITE_KEY
npx firebase-tools deploy --only hosting:jpstudy --project jpstudy-v2
```

3. Verified primary and legacy hosts with `curl -I`.
4. Queried Firebase Hosting channel metadata for both sites.
5. Patched the remaining generic deploy command in `docs/FIREBASE_SECURITY_CHECKLIST.md`.

## Result

- Primary deploy succeeded: `hosting[jpstudy]` release complete.
- Primary `https://jpstudy.web.app` returned `200` with CSP, XFO, Referrer-Policy, Permissions-Policy, and `Last-Modified: Thu, 14 May 2026 14:29:35 GMT`.
- Legacy `https://jpstudy-v2.web.app/?post-phase5=1` returned `404`.
- Firebase channel metadata:
  - `jpstudy` live release type: `DEPLOY`, release time `2026-05-14T14:29:35.567Z`.
  - `jpstudy-v2` live release type: `SITE_DISABLE`, release time `2026-05-14T13:51:20.135Z`.
- `.firebaserc` and `firebase.json` stayed primary-only after cleanup.
- `docs/FIREBASE_SECURITY_CHECKLIST.md` had one stale command, `firebase deploy --only hosting`; it is now `firebase deploy --only hosting:jpstudy`.

## Interpretation

The manual release path is now coherent for the active target. The process is not yet CI-integrated: no GitHub deploy workflow, live route smoke, Lighthouse budget, or notification policy is proven.

## Decision

Use the manual primary-only release path during onboarding phases. Do not add deploy automation until Q8 has a secrets policy and live smoke contract.
