# D8.Q8.2 Analysis - Firebase API Key Restriction

## E8.4 Result

The Firebase web API key is already protected by HTTP referrer restriction for Identity Toolkit. Fake-referrer probes to both `accounts:signUp` and `accounts:lookup` returned `403` with `API_KEY_HTTP_REFERRER_BLOCKED`.

Allowed hosting-origin probes to `accounts:lookup` returned `400 MISSING_ID_TOKEN`, which means the request passed the key/referrer layer and then failed normal Auth endpoint validation.

## Hypothesis Update

1. `H8.2.1` supported. Fake referrer `https://evil.example/probe` is blocked at the API-key layer.
2. `H8.2.2` falsified for the web key. The fake referrer did not reach endpoint validation.
3. `H8.2.3` falsified for Identity Toolkit web-key use. The key is not broadly usable from arbitrary HTTP referrers in this probe.
4. `H8.2.4` partially supported. The repository security checklist covered App Check/Auth hardening, but did not explicitly record API-key referrer restrictions or a repeatable verification command before E8.4.

## Beta Impact

This reduces one D8 launch risk: the public web Firebase key is not currently open to arbitrary browser referrers for Auth API calls. It does not replace App Check, Auth domain review, quota monitoring, or Storage rules.

## Red Team

Steelman opposite: this probe only verifies the web key against Identity Toolkit with supplied HTTP referrers. It does not prove all Google APIs are restricted, does not verify Android/iOS key package restrictions, and does not prove the Firebase Console allowed-referrer list is minimal. A full launch gate should still inspect GCP Console restrictions manually and keep this probe in the security checklist.

## Decision

Mark D8.Q8.2 as measured and currently passing for the web Auth key. Next action is documentation hardening:

1. Add an API-key restriction section to `docs/FIREBASE_SECURITY_CHECKLIST.md`.
2. Record expected allowed hosts: `jpstudy-v2.web.app` and `jpstudy.web.app`.
3. Keep App Check as a separate control; do not treat referrer restriction as abuse/DDOS protection.
