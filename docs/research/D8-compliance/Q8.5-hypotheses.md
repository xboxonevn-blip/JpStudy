# D8.Q8.5 Hypotheses - CI/CD Gates

Question: Does the repository CI/CD cover PR analysis/tests, storage rules, web build, post-deploy smoke, Lighthouse, and failure notification?

Commit: `aa5a2c441301bc43f7770325b41f0f4a7dd02bd7`

## Hypotheses

### H8.5.1 - CI is mostly absent

Prediction: `.github/workflows` has either no workflow or only a narrow guard, so PRs do not consistently run Flutter analyze/test/build.

Expected effect size: High release risk; add a baseline CI workflow before beta.

### H8.5.2 - CI has local quality gates but not release-parity build flags

Prediction: Existing workflow runs `flutter analyze`, `flutter test`, and `flutter build web`, but does not use release/base-href/App Check dart-define flags from `SHIPPING.md`.

Expected effect size: Medium release risk; CI can catch source/test failures but not exact release build drift.

### H8.5.3 - CI lacks post-deploy live smoke and Lighthouse gates

Prediction: No workflow runs route smoke or Lighthouse against `jpstudy-v2.web.app` / `jpstudy.web.app` after deploy.

Expected effect size: High D4/D7 carry-forward risk; stale live channels can pass local CI.

### H8.5.4 - Firebase Storage rules are covered

Prediction: Existing Node job runs `npm ci` and `npm run test:storage-rules`.

Expected effect size: Reduces cloud-backup rule regression risk.

### H8.5.5 - Failure notification is implicit only

Prediction: Failures show in GitHub Checks, but no Discord/Slack/email notification workflow exists.

Expected effect size: Medium process risk; silent broken deploy gates can linger unless branch protection or notifications are configured.
