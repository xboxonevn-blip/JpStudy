# D8.Q8.6 Analysis - Release Process Integration

## Verdict

Release process integration is good enough for manual onboarding-gate deploys, but not for automated beta release.

The chosen path is now explicit: build with the App Check site key, deploy only `hosting:jpstudy`, verify primary `200`, and verify legacy `jpstudy-v2` remains disabled. The main remediation was removing the last active-doc instruction that could deploy every Hosting target.

## Findings

1. Primary-only deploy works.
   - `npx firebase-tools deploy --only hosting:jpstudy --project jpstudy-v2` succeeded.
   - Firebase channel metadata shows `jpstudy` live release type `DEPLOY` at `2026-05-14T14:29:35.567Z`.

2. Legacy Hosting stayed closed after the clean deploy.
   - `https://jpstudy-v2.web.app/?post-phase5=1` returned `404`.
   - Firebase channel metadata shows `jpstudy-v2` live release type `SITE_DISABLE` at `2026-05-14T13:51:20.135Z`.

3. A real process hazard was found and patched.
   - `docs/FIREBASE_SECURITY_CHECKLIST.md` still said `firebase deploy --only hosting`.
   - That command is unsafe because it can deploy all configured Hosting targets if stale local target config returns.
   - The checklist now says `firebase deploy --only hosting:jpstudy` and includes primary/legacy curl checks.

4. Automation is still intentionally incomplete.
   - CI builds and smokes local release output.
   - CI does not deploy, smoke the live URL, run Lighthouse, notify failures, or prove branch protection.
   - This is acceptable for the current manual rollout because Firebase deploy credentials and App Check site key handling are not yet specified for GitHub Actions.

## Hypothesis Results

- H8.6.1 supported: active release docs are primary-only after the checklist patch.
- H8.6.2 supported: primary-only deploy did not re-enable legacy Hosting after config cleanup.
- H8.6.3 supported: post-deploy must check both primary and legacy because a stray redirect target briefly reactivated the legacy site earlier.
- H8.6.4 supported: do not add CI deploy yet.
- H8.6.5 supported: generic `firebase deploy --only hosting` is unsafe in this repo.

## Release Contract

Manual web release for the onboarding phases:

```powershell
flutter analyze lib test
flutter test
flutter build web --release --base-href=/ --dart-define=JPSTUDY_RECAPTCHA_SITE_KEY=$env:JPSTUDY_RECAPTCHA_SITE_KEY
npx firebase-tools deploy --only hosting:jpstudy --project jpstudy-v2
curl.exe -I https://jpstudy.web.app
curl.exe -I "https://jpstudy-v2.web.app/?legacy-disabled-check=1"
npx firebase-tools hosting:channel:list --site jpstudy --project jpstudy-v2 --json
npx firebase-tools hosting:channel:list --site jpstudy-v2 --project jpstudy-v2 --json
```

## Product Impact

The live channel is no longer ambiguous: beta UAT should target only `https://jpstudy.web.app`. The legacy URL is an operational check, not a product URL.

Remaining beta release risks are now narrower: live route smoke, Lighthouse/trace, App Check telemetry/enforcement, and error monitoring.
