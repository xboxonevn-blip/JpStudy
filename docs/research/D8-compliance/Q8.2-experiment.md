# D8.Q8.2 Experiment - Firebase API Key Restriction

## Experiment E8.4

Question: Does the current web Firebase API key reject unauthorized browser referrers?

Method:

- Read the actual web Firebase key and project/app identity from `lib/firebase_options.dart`.
- Read hosting targets from `.firebaserc` and `firebase.json`.
- Read security docs for an explicit API-key restriction gate.
- Send Identity Toolkit probes with fake `Origin` and `Referer`.
- Treat `403` with referrer/API-key wording as evidence for restriction; treat endpoint-level `400` as evidence that the fake referrer was not blocked at the key layer.

Probe commands:

```text
$headers = @{
  Origin = 'https://evil.example'
  Referer = 'https://evil.example/probe'
  'Content-Type' = 'application/json'
}

Invoke-WebRequest `
  -Uri 'https://identitytoolkit.googleapis.com/v1/accounts:signUp?key=<web-api-key>' `
  -Method Post `
  -Headers $headers `
  -Body '{"returnSecureToken":true}' `
  -UseBasicParsing

Invoke-WebRequest `
  -Uri 'https://identitytoolkit.googleapis.com/v1/accounts:lookup?key=<web-api-key>' `
  -Method Post `
  -Headers $headers `
  -Body '{}' `
  -UseBasicParsing
```

Expected information gain: high. A single fake-referrer probe distinguishes a key-layer block from normal Auth endpoint validation.

Limitations:

- This cannot change restrictions; GCP Console is manual.
- A command-line request is not a full browser-origin simulation, but Google API key restriction errors still surface on HTTP requests when the supplied referrer is blocked.
- A non-403 result does not prove exploitability by itself; it only means HTTP referrer restriction is not confirmed by this probe.
