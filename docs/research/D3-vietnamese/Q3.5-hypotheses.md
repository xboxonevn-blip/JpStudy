# D3.Q3.5 Hypotheses - ARB Migration Recommendation

Commit: `8387e444`

## Question

Is migration from the current `AppLanguage` enum-extension model to Flutter generated localizations (`flutter gen-l10n`, ARB files, ICU messages) worth doing before the 100-learner beta?

## Hypotheses

H3.5.1: Full ARB migration is not launch-worthwhile before beta because `AppLanguage` usage is spread across more than `100` files and more than `5,000` enum-locale branches.

H3.5.2: The app already has partial Flutter localization infrastructure (`flutter_localizations`, `MaterialApp.locale`, delegates), but no generated ARB workflow (`l10n.yaml`, `.arb`) exists.

H3.5.3: ICU/plural benefits are real for English/Japanese quality but are not the highest D3 risk for Vietnamese learners; glossary consistency and hardcoded-copy ownership will produce more launch value per hour.

H3.5.4: A hybrid path is cheaper: keep `AppLanguage` for beta, extract high-risk feature copy into feature-copy modules, and pilot ARB only after top terminology defects are stable.
