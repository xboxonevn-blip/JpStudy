# D3.Q3.2 Analysis - Hardcoded Vietnamese Outside AppLanguage

## E3.2 Result

Vietnamese UI/data copy is not centralized. The current audit found `1,893` Vietnamese-diacritic lines in Dart files outside `lib/core/app_language.dart`, excluding research helper code.

## Hypothesis Updates

- H3.2.1 supported. Hardcoded Vietnamese outside `app_language.dart` is far above the `20`-line falsifier.
- H3.2.2 partially supported. At least `396` matches are explicit local localization patterns (`AppLanguage.vi =>` or `_tr(...)`) across `28` files, but the full `1,893` includes data/model constants and copy helper files too.
- H3.2.3 supported. The top ten files account for `757` hits, with the biggest clusters in radical/kanji/vocab/custom-deck modules.

## What This Means

- `app_language.dart` is not the single editorial surface.
- A full ARB migration would be expensive because local copy helpers and data constants are already widespread.
- The first practical cleanup is an ownership map: central UI chrome in `app_language.dart`; domain data/copy files allowed only with explicit rationale.

## Red Team

Not every hit is wrong. `radical_item.dart` and kanji/domain copy may be content data rather than UI translation debt. The audit measures editorial surface area, not severity.

## Action

Before changing strings, classify the top ten files into:

1. UI chrome that should move to `app_language.dart`.
2. Domain content that should stay near the feature/data model.
3. Temporary/local helper copy that should be consolidated.
