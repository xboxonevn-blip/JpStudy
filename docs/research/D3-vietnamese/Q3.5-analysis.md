# D3.Q3.5 Analysis - ARB Migration Recommendation

## Recommendation

Do not do a full ARB migration before the 100-learner beta.

Use a staged path:

1. Keep `AppLanguage` and `appLanguageProvider` for beta.
2. Continue extracting high-risk feature copy into feature-copy modules (`vocab_copy.dart`, `kanji_copy.dart`, future `grammar_copy.dart`, etc.).
3. Add glossary-backed tests for the highest-traffic Vietnamese strings.
4. After beta signal, pilot ARB on global shell/settings/auth copy only.
5. Migrate feature copy to ARB only when translator workflow becomes the bottleneck.

## Evidence

- `app_language.dart` is `7,561` lines.
- It has `680` localized returns per locale and no blank/TODO returns.
- There are `1,893` Vietnamese lines outside `app_language.dart` after excluding research helper code.
- `AppLanguage` appears in `140` lib files.
- `appLanguageProvider` appears in `111` lib files.
- `AppLanguage.en/vi/ja` appears `5,219` times.
- Locale switch branches/arrows appear `5,115` times.
- `flutter_localizations` is already a dependency and `MaterialApp.router` already sets `locale`, supported locales, and material delegates.
- There is no `l10n.yaml` and `0` `.arb` files.

## Hypothesis Updates

- H3.5.1 supported. The call-site surface is too broad for a low-risk beta-prep migration.
- H3.5.2 supported. Flutter localization hooks exist, but generated localization does not.
- H3.5.3 supported. D3.Q3.4 found `8/100` raw-English-term warnings and no punctuation/tone/mojibake warnings; current Vietnamese risk is wording ownership, not ICU.
- H3.5.4 supported. Feature-copy modules match existing patterns and avoid `BuildContext` leakage into providers/models/export services.

## Effort Estimate

Full ARB migration before beta:

- Create `l10n.yaml`, ARB files, generated localization setup: `0.5-1` day.
- Convert `680 x 3` app-language returns into ARB keys and ICU syntax: `2-3` days.
- Rewrite or bridge `5,219` enum-locale references across `140` files: `4-6` days.
- Refactor providers/models/export services that currently accept `AppLanguage` and cannot naturally read `BuildContext`: `2-4` days.
- QA all three locales across navigation, settings, learning, test, search, and export flows: `2-3` days.

Estimated full migration: `10-17` engineer-days, with high regression risk.

Staged beta path:

- Extract/correct top feature-copy clusters and add glossary tests: `1-3` days.
- Add ARB pilot for global shell only after Q3.6 plural audit: `1-2` days.
- Keep `AppLanguage` as the app language/domain signal until provider/model boundaries are redesigned.

Estimated staged path before beta: `2-5` engineer-days, lower risk.

## Red Team

- ARB would improve translator workflow and reduce long-term copy drift. Delaying it increases later migration debt.
- Current feature-copy modules still require engineers to edit Dart for translation fixes, which is not ideal for nontechnical Vietnamese editors.
- ICU matters for English/Japanese polish and may uncover bugs in Q3.6.
- A tiny ARB pilot could be useful soon, but a full migration now would consume time better spent on content quality, persona UAT, and launch compliance.

## Action

Do Q3.6 next. Use it to identify actual plural/ICU defects. If Q3.6 finds high user-facing English/Japanese plural breakage, do a small ARB pilot for those messages. Otherwise keep the staged feature-copy approach through beta.
