# D5.Q5.1 Analysis - FSRS Correctness

## E5.1 Result

Current SRS scheduling is not FSRS-6-conformant. The app uses a legacy `17`-parameter FSRS-like service, lacks persisted learning/relearning state, and schedules new-card `Again`, `Hard`, and `Good` ratings hours or days later instead of the FSRS scheduler's minute-scale learning steps.

## Findings

1. `H5.1.1` supported. Local `_w` has `17` weights; current `fsrs-rs` defaults have `21` weights and an FSRS-6 decay value.
2. `H5.1.2` strongly supported. First-review intervals diverge most on new cards:
   - `Again`: local `576` min vs FSRS-6 scheduler `1` min.
   - `Hard`: local `864` min vs FSRS-6 scheduler `5.5` min.
   - `Good`: local `3,456` min vs FSRS-6 scheduler `10` min.
   - `Easy`: local `8,352` min vs FSRS-6 scheduler `11,520` min.
3. `H5.1.3` supported. The old tests pass because they check monotonic direction, clamps, and due-date ordering. They did not pin FSRS-6 defaults or learning steps.
4. `H5.1.4` partially supported. `0.9` retention is a normal target, but it is only a global constant here; no optimizer or cohort calibration was found.
5. `H5.1.5` supported. `SrsState` stores `box`, `repetitions`, `ease`, `stability`, `difficulty`, confidence, and timestamps. It does not persist FSRS card `state` or `step`, so default learning/relearning steps cannot be faithfully represented.

## Production Impact

This is a beta blocker for measured learning outcomes. A learner who presses `Again` on a brand-new card should see it again immediately in the learning loop; local scheduling can delay it by `9.6` hours. A learner who presses `Good` on a new card gets `2.4` days locally, while the reference scheduler keeps it in learning for `10` minutes before graduation. That can inflate apparent progress while lowering actual retention.

## Red Team

Steelman opposite: JpStudy may intentionally use a simplified scheduler, not literal Anki/FSRS-6, to avoid noisy minute-level reviews in a casual mobile app. That is a valid product choice only if it is named and measured as a custom scheduler. It does not satisfy Q5.1's FSRS-6 correctness bar, and the current tests give no retention evidence that the simplification works for Vietnamese JLPT learners.

## Decision

Do not claim FSRS-6 correctness or production-ready SRS yet. Treat current code as a legacy FSRS-like scheduler.

## Action

Next implementation should be a narrow `Q5.1b` fix plan:

1. Add failing spec tests for new-card `Again`, `Hard`, and `Good` learning-step intervals.
2. Add persisted FSRS card `state` and `step` with migration for existing `srs_state`, `kana_srs`, and `kanji_srs` equivalents if needed.
3. Decide product policy for minute-level reviews before changing live scheduling.
4. Run synthetic-user retention simulation after the scheduler change.
