# D5.Q5.5 Analysis - Mistake Review and Ghost Algorithm

## E5.5 Result

Mistake review is useful as a lightweight repair queue, but it is not a coherent learning scheduler. Vocab/kanji mistakes use a global mistake bank with a correct-streak counter and 1-3-7 display buckets. Grammar also has `ghostReviewsDue` in SRS state, plus an older attempt-answer ghost path. The split creates stale-ghost risk.

## Findings

1. `H5.5.1` supported. `user_mistakes` stores `type`, `itemId`, `wrongCount`, `lastMistakeAt`, and context fields. There is no persisted next-review schedule. 1-3-7 due buckets are computed from `lastMistakeAt` at read time.
2. `H5.5.2` supported. New mistakes are inserted with `wrongCount = MistakeDao.requiredCorrectStreak`, currently `2`. `markCorrect()` decrements until `wrongCount <= 1`, then removes on the next correct mark.
3. `H5.5.3` supported. Grammar ghosts use two systems: `grammar_srs_state.ghostReviewsDue` for current providers, and `AttemptAnswer` rows in `LessonRepository.fetchGrammarGhosts()` / `markGrammarAsMastered()`. These can drift.
4. `H5.5.4` supported. `GhostPracticeScreen` answer selection changes local score/feedback, but does not call `GrammarRepository.recordReview()` or `MistakeRepository.markCorrect()`. The "mark mastered" action calls the old `AttemptAnswer` deletion path, not the `ghostReviewsDue` source used by the screen.
5. `H5.5.5` supported. Repeated wrong answers increase `wrongCount`; priority scales by `wrongCount * 10`, but there is no remediation branch for high repeated failure.

## Production Impact

For vocab and kanji, the mistake bank is understandable: get it wrong, require two clean passes, show it again after 1/3/7-day age buckets. It is not FSRS, but it is explainable.

For grammar, the ghost experience is high risk. A learner can enter a grammar ghost practice session, answer correctly, see completion UI, and still keep the visible ghost because the quiz did not clear `ghostReviewsDue`. The explicit "mark mastered" path also targets old attempt-answer data, so it may not clear the visible provider either. This can make the app feel broken and create a false "mistake loop".

## Red Team

Steelman opposite: ghost practice may be intended as low-stakes drilling, with `mark mastered` as manual escape. But the UI labels it as practice for ghosts, invalidates ghost providers on completion, and shows success feedback. That strongly implies state should change.

## Decision

Do not treat current grammar ghost completion as reliable remediation. Treat vocab/kanji mistake bank as a simple repair queue, and grammar ghost state as needing source-of-truth unification before production claims.

## Action

Next implementation should be `Q5.5b`:

1. Pick one grammar ghost source of truth, preferably `grammar_srs_state.ghostReviewsDue` plus global `user_mistakes` context.
2. Make `GhostPracticeScreen` call `GrammarRepository.recordReview()` on each answer and clear/decrement the matching mistake row on correct answers.
3. Make "mark mastered" clear the same `ghostReviewsDue` field used by `grammarGhostsProvider`.
4. Add tests proving a correct ghost-practice answer removes or decrements the visible ghost.
5. Add an escalation policy for high `wrongCount`, e.g. show explanation/prerequisite/hint-first mode instead of only increasing priority.
