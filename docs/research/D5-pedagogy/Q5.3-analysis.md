# D5.Q5.3 Analysis - XP and Level Formula

## E5.3 Result

XP is not governed by one product policy. Level calculation is deterministic and well-tested, but reward generation is fragmented across modules, visible XP is sometimes not global XP, and there is no daily cap or burnout guard.

## Findings

1. `H5.3.1` supported. Independent XP formulas exist for vocab SRS review, learn questions, test sessions, flashcards, match games, kanji dash, daily-session completion, weekly challenge bonuses, and achievement bonus labels.
2. `H5.3.2` supported. Learn summary displays `+${session.totalXP} XP` and `LearnSessionService` stores `learn_sessions.xpEarned`, but learn completion does not call `recordStudyActivity()`. Flashcard summary also displays calculated XP without a discovered global write path. Achievement `bonusXP` is displayed by achievement widgets but is not consistently written to global `user_progress`.
3. `H5.3.3` supported. No `daily xp cap`, `xp cap`, `burnout`, `diminishing`, or `throttle` policy was found. `recordStudyActivity()` adds any positive delta.
4. `H5.3.4` supported. The actual level curve is an increasing-gap ladder: `0`, `100`, `250`, `450`, `700`, ...; tests pin this. The top comments in `LevelCalculator` still mention simple quadratic progression and conflicting ranges, which should be cleaned up.
5. `H5.3.5` supported. XP mostly rewards correctness/speed/score, not retention quality. Vocab SRS gives XP even for `Again`, while wrong learn/test answers earn `0`; games can produce score-based XP. There is no calibration tying XP to retention or mastery.

## Production Impact

XP currently works as local motivation copy, but it is unsafe as a unified progress signal. A learner may see XP in Learn or Flashcards that does not move dashboard level, while a test or game does. This can make the app feel inconsistent and can distort funnel/retention analysis if "today XP" is treated as "study effort".

No daily cap may be acceptable for beta if XP is only a cosmetic reward. It is risky if XP drives leaderboard, weekly challenges, or level-up pressure, because high-volume game/test sessions can dominate XP more than SRS quality.

## Red Team

Steelman opposite: module-specific XP lets each activity feel natural; not all XP needs to be global. That could be valid if labels distinguish local session XP from account XP. Current UI just says `+XP`, so users and telemetry will read it as one currency.

## Decision

Do not interpret `todayXp` or level as a normalized learning-effort metric. Treat global XP as a partially populated gamification counter.

## Action

Next implementation should be `Q5.3b`:

1. Create a central `XpPolicy` with named sources and expected global/local behavior.
2. Route Learn, Flashcards, achievements, and grammar review through a clear global write decision.
3. Add tests that every visible account-level `+XP` updates `user_progress`.
4. Decide whether beta needs a daily XP cap or diminishing returns before enabling competitive leaderboard framing.
5. Update `LevelCalculator` comments to match the tested increasing-gap curve.
