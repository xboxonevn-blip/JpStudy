# D5.Q5.2 Analysis - Streak Rules

## E5.2 Result

Current streak logic is simple and understandable, but not production-ready as a cross-skill 30-day retention signal. It uses device-local midnight, has no freeze/repair/grace policy, can miss global credit for grammar SRS review, and has no uniqueness guard on `user_progress.day`.

## Findings

1. `H5.2.1` supported. `recordStudyActivity`, `recordReview`, `fetchProgressSummary`, daily plan, daily session progress, and reminder dedupe use `DateTime.now()` and local date keys. There is no configurable day boundary. This differs from Anki's explicit "next day starts at" setting, where default day rollover is `4AM` in the current timezone.
2. `H5.2.2` supported. If there is no today row, `fetchProgressSummary()` carries yesterday's streak; if the latest row is older than yesterday, streak is `0`. There is no freeze, repair, grace period, or subscription reinstatement rule. Duolingo's published streak design includes Streak Freeze slack and repair tooling for outages.
3. `H5.2.3` supported. Vocab review calls `LessonRepository.saveTermReview()` then global `recordReview()`, and many game/test/daily-session paths call `recordStudyActivity()`. Grammar review uses `GrammarRepository.recordReview()` and only updates `grammar_srs_state.streak`; it does not update global `user_progress`.
4. `H5.2.4` supported. `UserProgress` has no unique key on `day`, and `_ensureProgressRow()` performs SELECT-then-INSERT. Duplicate same-day rows can happen under first-activity races or imported/legacy data, which can make `getSingleOrNull()` unsafe.
5. `H5.2.5` supported. Existing tests pass but do not pin timezone/day-boundary, freeze/repair, all-skill credit, or duplicate-day behavior.

## Production Impact

For the North Star, streak is not the main metric, but it affects session quality and daily retention. A learner can complete grammar review without extending the global streak, while a weekly challenge or daily-session completion can extend it through generic XP. This makes streak less trustworthy as a retention proxy and less fair across personas who mostly study grammar or reading.

The no-freeze policy is not necessarily wrong, but it is currently implicit. For a 30-day beta, a hard one-missed-day reset may punish otherwise healthy learners, especially busy P2/P3 users. If the product wants Anki-like rigor, expose a day-boundary policy. If it wants Duolingo-like habit protection, add a limited freeze/grace system.

## Red Team

Steelman opposite: a strict local-midnight streak is easier to explain and avoids scope creep. Grammar review's local `streak` field may intentionally mean grammar-specific streak, not global streak. That still leaves two issues: the UI displays a global streak, and the code does not define which activities count. Hidden policy is the core defect.

## Decision

Do not use current streak as a high-confidence learning or retention signal. Treat it as a lightweight gamification display until policy and write paths are normalized.

## Action

Next implementation should be `Q5.2b`:

1. Add a `StreakPolicy` helper with injectable clock and configurable day boundary.
2. Add tests for today/yesterday/two-days-ago, near-midnight study, and duplicate same-day rows.
3. Add a unique index or upsert path for `user_progress.day`.
4. Route all learner-facing study completions through one global activity recorder, or explicitly document which modes do not count.
5. Decide no-grace vs limited freeze before beta copy or launch metrics use streak.
