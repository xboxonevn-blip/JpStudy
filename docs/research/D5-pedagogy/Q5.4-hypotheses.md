# D5.Q5.4 Hypotheses - Onboarding Fork Logic

## H5.4.1 - Onboarding captures only level and broad goal

Prediction: onboarding persists `StudyLevel` and `StudyGoal`, but does not capture exam date, time budget, study intensity, use case, accessibility preference, or learner context. Effect size: persisted onboarding prefs are exactly completed/level/goal.

Falsifier: onboarding stores a richer profile used by later routing or planning.

## H5.4.2 - Goal selection barely changes the first real path

Prediction: after completion, only `StudyGoal.writing` triggers an immediate route change; `jlpt` and `reading` both land on the same home path and rely on generic continue/daily-plan logic.

Falsifier: each goal routes to a distinct first session or dashboard lane.

## H5.4.3 - First-win preview is always vocab, not goal-specific

Prediction: the first-win page loads `getVocabByLevel(level.shortLabel)` and builds a vocab meaning multiple-choice item regardless of selected goal. If a level has fewer than three items, the fallback preview is hardcoded N5 vocab.

Falsifier: JLPT, reading, and writing goals get separate preview tasks, or fallback content is level/goal scoped.

## H5.4.4 - Daily plan ignores onboarding goal and intensity

Prediction: daily-plan generation depends on due counts, mistakes, and current level, with fixed new-content batches (`5` vocab, `3` grammar, `3` kanji) when review debt is low. It does not read `studyGoalProvider`, exam deadline, or daily time target.

Falsifier: daily plan changes workload and lane order by onboarding goal or time budget.

## H5.4.5 - Onboarding telemetry cannot explain persona fork fit

Prediction: `onboarding_completed` logs only `level` and `goal`, so beta analysis cannot separate cramming, travel/fun, group study, accessibility, or news-reading intent.

Falsifier: onboarding telemetry includes intent/time/accessibility/context fields and downstream route attribution.
