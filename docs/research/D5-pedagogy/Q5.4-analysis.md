# D5.Q5.4 Analysis - Onboarding Fork Logic

## E5.4 Result

Onboarding is clean enough for a narrow "choose JLPT level, pick a broad goal, answer a preview vocab item" entry flow, but it is not a production-grade fork engine for mixed beta personas. It stores level and one of three goals, then most downstream surfaces use generic due-count, mistake-count, XP, and level logic.

## Findings

1. `H5.4.1` supported. Persisted onboarding state is `onboarding.completed`, `onboarding.level`, and `onboarding.goal`. There is no exam date, daily minutes, cramming intensity, travel/fun intent, group-study intent, text-size setting, or news/current-world reading preference.
2. `H5.4.2` supported. `StudyGoal.writing` redirects to Hiragana foundations after completion; `StudyGoal.jlpt` and `StudyGoal.reading` do not trigger distinct first-session routes.
3. `H5.4.3` supported. First-win preview always calls `getVocabByLevel(level.shortLabel)` and renders a vocab meaning multiple-choice question. If the selected level has fewer than three vocab items, fallback preview content is hardcoded N5.
4. `H5.4.4` supported. Daily plan reads dashboard due/mistake counts and current level, then adds fixed new-content batches when total due is under `40`. It does not watch `studyGoalProvider` or a time/intensity profile.
5. `H5.4.5` supported. `onboarding_completed` telemetry carries only `level` and `goal`, so analytics cannot explain whether a learner wanted cramming, casual travel, news reading, accessibility help, or group study.

## Production Impact

This is acceptable for a controlled N5 habit-building pilot where "level + broad goal" is enough to get started. It is not enough for the D4 mixed-persona beta: P3 needs a cramming/intensity fork, P4 needs travel/fun plus accessibility choices, and P5 needs an advanced reading/news path. The current flow can record those learners as "N2/JLPT", "N4/reading", or "N1/reading", but it cannot route or measure the product promise they actually care about.

The strongest existing piece is persistence: `appInitProvider` now loads level/goal before direct routes finish building. That fixes state trust better than adding more onboarding questions would. The missing layer is a small profile/policy object that turns onboarding answers into first-session routing, daily-plan workload, and telemetry fields.

## Red Team

Steelman opposite: keeping onboarding short reduces drop-off, and over-segmenting early users can create fake personalization. That is a valid beta constraint. The defect is not that onboarding is short; the defect is that the app copy implies multiple study intents while downstream behavior mostly ignores the chosen intent.

## Decision

Treat current onboarding as a minimal account setup, not a pedagogical fork. Do not claim persona-specific personalization or use goal-level telemetry as proof that the app served those goals.

## Action

Next implementation should be `Q5.4b`:

1. Add an `OnboardingProfile` with `level`, `goal`, optional `dailyMinutes`, optional `examDate`, and optional `intentTag`.
2. Keep default flow short: ask extra questions only when the selected goal needs them, e.g. JLPT exam date/intensity or reading news/casual choice.
3. Route first session by profile: JLPT to coach/mock path, reading to immersion level path, writing to kana/kanji foundations.
4. Make daily plan accept a profile-derived time budget instead of fixed-only batch sizes.
5. Extend `onboarding_completed` telemetry with non-sensitive profile tags needed for beta analysis.
