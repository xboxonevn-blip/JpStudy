# D5.Q5.1 Hypotheses - FSRS Correctness

## H5.1.1 - Local scheduler is a legacy FSRS variant, not FSRS-6

Prediction: `FsrsService` uses fewer than the FSRS-6 `21` default parameters and omits the FSRS-6 decay parameter. Effect size: local parameter count is `17`, and current FSRS-6 source has `21`.

Falsifier: local `FsrsService` has `21` parameters including the FSRS-6 decay value and uses the current power forgetting curve.

## H5.1.2 - New-card scheduling is the largest correctness risk

Prediction: local first-review intervals apply the long-term retention interval immediately, while FSRS-6 scheduler defaults keep new cards in short learning steps. Effect size: local `Again`, `Hard`, or `Good` first intervals differ from FSRS-6 defaults by more than `100x`.

Falsifier: local first-review `Again` is about `1` minute, `Hard` about `5.5` minutes, and `Good` about `10` minutes.

## H5.1.3 - Existing tests prove invariants, not FSRS-6 conformance

Prediction: focused SRS tests pass even while the audit reports FSRS-6 divergence because tests assert monotonic properties, clamps, and due-date ordering rather than pinned FSRS-6 reference outputs.

Falsifier: tests fail on FSRS-6 reference-value mismatches or already pin first-review learning-step behavior.

## H5.1.4 - Retention target is reasonable but uncalibrated

Prediction: `defaultRetention = 0.9` matches common FSRS defaults, but there is no project-specific optimization from Japanese/Vietnamese learner review logs. Effect size: no local code path computes optimal retention or parameters from review logs.

Falsifier: repository contains a review-log optimizer or synthetic-user calibration that adjusts parameters/retention per learner cohort.

## H5.1.5 - Schema cannot represent FSRS learning/relearning state

Prediction: `srs_state` stores stability/difficulty and SM-2-like fields, but not FSRS card `state` or `step`. Effect size: no persisted learning/relearning state means the app cannot faithfully implement default FSRS learning or relearning steps.

Falsifier: Drift schema persists card state and step, and repository update paths use those fields in scheduling.
