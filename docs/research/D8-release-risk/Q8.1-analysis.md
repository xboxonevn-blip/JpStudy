# D8.Q8.1 Analysis - Release Risk Inventory

## E8.1 Result

Current release risk is not a single blocker. It is a stack: live channels are behind local source, full-CI confidence was briefly blocked by a stale route smoke assertion, web App Check is omitted by the basic shipping command, security docs disagree with the active CSP config, GA4 export tables are not present yet, and D6/D7 product-readiness risks remain.

## E8.6 Remediation Update

Two release-doc drift items from E8.1 are now patched:

- `SHIPPING.md` requires `JPSTUDY_RECAPTCHA_SITE_KEY` in the web release build and makes Hosting target selection explicit.
- `docs/FIREBASE_SECURITY_CHECKLIST.md` reflects that `firebase.json` configures CSP for both Hosting targets and requires header verification for both live URLs.

Still open: no deploy was attempted, live channels may still be behind local source, App Check telemetry/enforcement remains a Console check, and D7 needs live performance smoke after the chosen target is refreshed.

E8.6 note: item 4 has been remediated in repository docs, but not validated through a fresh deploy.

## Beta Impact

Do not use either live hosting URL as proof of current local readiness until a fresh deploy/channel decision is made. D4 route-localization failures were plausibly measured against stale hosting. E8.6 reduced release-doc drift, but a fresh deploy alone is still not enough: App Check telemetry/enforcement, D4 live route probes, and D7 reproducible perf budget remain required.

Highest-risk release items:

1. Channel freshness: both live sites are older than current `HEAD`.
2. App Check deployment proof: docs now require the site key, but telemetry/enforcement has not been verified after a fresh deploy.
3. Live header proof: docs now match local CSP config, but headers have not been rechecked after a fresh deploy.
4. Measurement gap: GA4 export dataset/tables still unavailable.
5. Product-readiness carry-forward: D6 accessibility/dark-mode risks and D7 first-load resource count remain unresolved.

## Red Team

Steelman opposite: focused smoke tests are green, release web build passed in D7, and Firebase Hosting has reasonable headers and SPA rewrites. That supports a controlled private beta deploy if the team accepts known measurement and product-polish caveats. It does not support public launch language, performance claims, dark-mode parity claims, or metric-grade NS reporting yet.

## Decision

Treat D8 as "release candidate needs hygiene pass," not "do not ship ever." The next release action should be:

1. Keep shipping/security docs aligned with CSP/App Check requirements when deploy targets or external endpoints change.
2. Run `flutter analyze`, `flutter test`, and `flutter build web --release --base-href=/`.
3. Build web with `--dart-define=JPSTUDY_RECAPTCHA_SITE_KEY=<site_key>` if web App Check is required for beta.
4. Make an explicit deploy target decision: `jpstudy-v2`, `jpstudy`, or both.
5. After deploy, rerun D4 direct-route probes and D7 perf smoke against the chosen live URL.

No deploy was attempted in this experiment.
